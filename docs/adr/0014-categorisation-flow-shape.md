# Categorisation flow is one composite endpoint with IBAN-exact `Counterparty` match, a last-used `Account` heuristic, and own-account pre-fill for self-transfers; rules engine is deferred

The **Categorisation flow** that turns one unmatched `BankTransaction` into one `JournalEntry` is exposed as a single composite endpoint `POST /api/bank-transactions/{id}/categorize` that accepts (a) either an existing `CounterpartyId` *or* a `NewCounterparty: { Name }` block (mutually exclusive, validated in FluentValidation), (b) an optional `LinkBankAccount: { Iban?, AccountNumber?, AccountHolderName? }` payload that creates a counterparty-owned `BankAccount` (`CounterpartyId` set, `AccountId` null, `CurrencyCode` null per ADR 0011) when the row's `CounterpartyAccountNumber` is not already mapped, and (c) the `JournalEntry` fields (`Date`, `Description?`, `Lines: [{ AccountId, Amount, Description? }]`). All three writes run inside a single `BeginTransactionAsync` / `Commit`, so a `Counterparty` or `BankAccount` never persists unless the `JournalEntry` also persists; the response is the created `JournalEntryOutput`.

**Counter-side pre-fill at flow open** is resolved in this order. (1) **Self-transfer recognition**: if the BT's `CounterpartyAccountNumber` resolves (via exact match on `BankAccount.Iban`) to a `BankAccount` whose `AccountId IS NOT NULL` — i.e. one of your own `Account`s — the flow recognises a *self-transfer in progress* and pre-fills the counter-side line with that own-`Account`, leaving `CounterpartyId` empty on submit (the resulting `JournalEntry` has `CounterpartyId = null`, every line on an own-`Account`, the bank-side line `Cleared`, the counter-side line `Uncleared` waiting for the sibling BT to attach — see ADR 0013). (2) **Counterparty resolution**: otherwise, the BT's `CounterpartyAccountNumber` is matched against `BankAccount.Iban` to identify a `Counterparty` (when the matched `BankAccount.CounterpartyId IS NOT NULL`), and a single query for the most recent `JournalEntry` carrying that `CounterpartyId` defines the suggested counter-side `Account`(s) and amounts; overrides become the de facto next default on the next iteration, without any persisted "default" column. Counterparty identification is a single exact-match query — no name-based matching, no fuzzy normalisation, no description rules. The bank-side `JournalLine` is created with `ReconciliationStatus = Cleared` (the bank already confirmed the row); counter-side lines stay `Uncleared`. Splits are N counter-lines whose `Amount` values sum to `−BT.Amount`; cross-currency lines are refused — every line's `Account` must have `CurrencyCode = BT.Money.CurrencyCode` until the multi-currency machinery lands.

**Self-transfer second-leg flow.** When the *other* side's `BankTransaction` later imports (e.g. the savings statement after the current-account statement already produced its self-transfer JE), the **Inbox** surfaces a one-click `Attach` action when the **Attach predicate** (ADR 0013) uniquely identifies the matching JE. The Categorisation flow, opened on that sibling BT, includes a **manual JE-picker fallback** as a third option alongside `[Attach to JE-N (matched)]` (shown when the predicate found exactly one match) and `[Create new JE]`: `[Pick a JE to attach to...]` opens a search box over `JournalEntry`s that satisfy the *structural* conditions of the Attach predicate (own-account-only lines, currency match, an available `Uncleared` slot on the right `Account`) but relaxes the date window — the user widens the search themselves. The picker exists because the strict predicate misses edge cases (off-by-a-week SEPA settlement, missed-IBAN parse) where the user knows the link exists; the manual path keeps Attach the canonical action for the sibling case rather than forcing the user back to Dismiss when auto-detection misses.

We rejected (a) **three separate SPA-orchestrated calls** (`POST /counterparties` → `POST /bank-accounts` → `POST /journal-entries`) because the writes are semantically one user intent — "I am categorising this bank row" — and a mid-sequence failure leaves the SPA with awkward partial state (a created `Counterparty` and a `BankAccount` linking it to the IBAN, with no `JournalEntry`) that the user has no UI affordance to discover or roll back; the composite endpoint earns its keep through atomicity and a single `Result` error surface for the whole flow. (b) **Extending `POST /api/journal-entries` with optional `NewCounterparty` and `LinkBankAccount` blocks** because it bloats a generic domain endpoint with import-flow concerns, conflates two distinct user activities (cash `JournalEntry` creation vs. categorisation of an imported row) into one input contract, and makes the OpenAPI surface harder to read; the categorisation endpoint is import-flow-shaped (path is `/bank-transactions/{id}/categorize`) and the generic `POST /api/journal-entries` stays narrow. (c) **Adding `Counterparty.DefaultAccountId` (nullable FK) in v1** because the "last-used" heuristic costs zero schema, self-corrects on every override (the new override is itself the new "last used"), and the only thing the explicit column buys — a stable, user-curated default that survives an erratic one-off — is speculative until real usage shows which counterparties have stable defaults vs. erratic ones; the column can be added in a future migration without touching existing data, and only when a concrete pain point demands it. (d) **A rules engine** (regex on description, amount-range matching, multi-condition rule precedence) for v1 because it is textbook premature scope: every personal-finance tool eventually grows one, but the right shape — which fields are matched, precedence rules, conflict resolution, edit UX — is impossible to design well without first having the manual-categorisation pain points in front of us; deferring lets the v1 manual categorisation surface tell us what the rules should look like. (e) **Atomic categorise-and-attach-second-BT in a single endpoint call** when both BTs of a self-transfer are in the Inbox at flow open, because it widens the `categorize` input contract with an "also attach these BTs" list that'll be empty or single-element in practice, introduces a chicken-and-egg validation problem (the secondary BT's Attach predicate runs against the JE being created, which doesn't exist yet at validation time), and the user-facing saving over a follow-up Inbox-hint click is one click for a rare both-in-Inbox case — the natural flow is categorise BT-A → BT-B's Inbox row immediately shows the hint → one click to attach. (f) **Name-based or fuzzy `Counterparty` matching on flow open** because the bank's `CounterpartyName` is structurally noisy ("AH ROTTERDAM ZUID 1234"), POS-payment IBANs commonly route to payment processors (Adyen, Worldline, iZettle) rather than to merchants, and any reasonable fuzzy logic will silently merge unrelated counterparties whose names happen to collide; exact IBAN match is the only key whose failure mode is "user has to pick a `Counterparty`", which is the correct failure mode for a flow that creates ledger entries. (g) **Persisting flow state server-side across steps** (a `CategorisationDraft` entity, multi-step wizard with intermediate writes) because the flow's decision tree is small enough to hold in the SPA's local state until the user submits, and a draft entity introduces lifecycle complexity (expiration, conflicts on re-open, GC) that yields no benefit for a per-row categorisation that completes in seconds. (h) **An Inbox-side manual JE-picker** as a parallel to the auto-hint, because the picker is a multi-step UX (search, filter, confirm) that belongs in the flow alongside the other options the user is weighing, not as a fourth Inbox-row affordance competing with the one-click Attach hint.
