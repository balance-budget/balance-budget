# Categorisation flow is one composite endpoint with IBAN-exact `Counterparty` match and a last-used `Account` heuristic; rules engine and self-transfer detection are deferred

The **Categorisation flow** that turns one unmatched `BankTransaction` into one `JournalEntry` is exposed as a single composite endpoint `POST /api/bank-transactions/{id}/categorize` that accepts (a) either an existing `CounterpartyId` *or* a `NewCounterparty: { Name }` block (mutually exclusive, validated in FluentValidation), (b) an optional `LinkBankAccount: { Iban?, AccountNumber?, AccountHolderName? }` payload that creates a counterparty-owned `BankAccount` (`CounterpartyId` set, `AccountId` null, `CurrencyCode` null per ADR 0011) when the row's `CounterpartyAccountNumber` is not already mapped, and (c) the `JournalEntry` fields (`Date`, `Description?`, `Lines: [{ AccountId, Amount, Description? }]`). All three writes run inside a single `BeginTransactionAsync`/`Commit`, so a `Counterparty` or `BankAccount` never persists unless the `JournalEntry` also persists; the response is the created `JournalEntryOutput`. Counterparty identification on flow open is a single exact-match query on the BT's `CounterpartyAccountNumber` against `BankAccount.Iban` — no name-based matching, no fuzzy normalisation, no description rules; when the matched `BankAccount` has `CounterpartyId` set, that `Counterparty` is pre-selected and the user can override. Counter-side `Account` pre-selection is a single query for the most recent `JournalEntry` carrying the resolved `CounterpartyId` — its non-bank-side `JournalLines` define the suggested `Account`(s) and amounts for the new entry; overrides become the de facto next default on the next iteration, without any persisted "default" column. The bank-side `JournalLine` is created with `ReconciliationStatus = Cleared` (the bank already confirmed the row); counter-side lines stay `Uncleared`. Splits are N counter-lines whose `Amount` values sum to `−BT.Amount`; cross-currency lines are refused — every line's `Account` must have `CurrencyCode = BT.Money.CurrencyCode` until the multi-currency machinery lands.

We rejected (a) **three separate SPA-orchestrated calls** (`POST /counterparties` → `POST /bank-accounts` → `POST /journal-entries`) because the writes are semantically one user intent — "I am categorising this bank row" — and a mid-sequence failure leaves the SPA with awkward partial state (a created `Counterparty` and a `BankAccount` linking it to the IBAN, with no `JournalEntry`) that the user has no UI affordance to discover or roll back; the composite endpoint earns its keep through atomicity and a single `Result` error surface for the whole flow. (b) **Extending `POST /api/journal-entries` with optional `NewCounterparty` and `LinkBankAccount` blocks** because it bloats a generic domain endpoint with import-flow concerns, conflates two distinct user activities (cash `JournalEntry` creation vs. categorisation of an imported row) into one input contract, and makes the OpenAPI surface harder to read; the categorisation endpoint is import-flow-shaped (path is `/bank-transactions/{id}/categorize`) and the generic `POST /api/journal-entries` stays narrow. (c) **Adding `Counterparty.DefaultAccountId` (nullable FK) in v1** because the "last-used" heuristic costs zero schema, self-corrects on every override (the new override is itself the new "last used"), and the only thing the explicit column buys — a stable, user-curated default that survives an erratic one-off — is speculative until real usage shows which counterparties have stable defaults vs. erratic ones; the column can be added in a future migration without touching existing data, and only when a concrete pain point demands it. (d) **A rules engine** (regex on description, amount-range matching, multi-condition rule precedence) for v1 because it is textbook premature scope: every personal-finance tool eventually grows one, but the right shape — which fields are matched, precedence rules, conflict resolution, edit UX — is impossible to design well without first having the manual-categorisation pain points in front of us; deferring lets the v1 manual categorisation surface tell us what the rules should look like. (e) **Auto-detecting the second side of a between-own-accounts transfer at flow open** (checking whether the BT's `CounterpartyAccountNumber` matches a `BankAccount` with `AccountId IS NOT NULL`) because it implies either an opinionated "this should be one `JournalEntry`, not two" UI branch that requires schema change (ADR 0013 fixes the JE↔BT cardinality at 1:1) or a clearing-account model — both expand v1 scope materially; v1 makes the user handle self-transfers manually and **Dismiss** the sibling row, accepting the manual step in exchange for not building auto-pairing UX twice. (f) **Name-based or fuzzy `Counterparty` matching on flow open** because the bank's `CounterpartyName` is structurally noisy ("AH ROTTERDAM ZUID 1234"), POS-payment IBANs commonly route to payment processors (Adyen, Worldline, iZettle) rather than to merchants, and any reasonable fuzzy logic will silently merge unrelated counterparties whose names happen to collide; exact IBAN match is the only key whose failure mode is "user has to pick a `Counterparty`", which is the correct failure mode for a flow that creates ledger entries. (g) **Persisting flow state server-side across steps** (a `CategorisationDraft` entity, multi-step wizard with intermediate writes) because the flow's decision tree is small enough to hold in the SPA's local state until the user submits, and a draft entity introduces lifecycle complexity (expiration, conflicts on re-open, GC) that yields no benefit for a per-row categorisation that completes in seconds.
