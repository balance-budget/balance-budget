# Development sample data is seeded at runtime via UseAsyncSeeding, rebuilt every startup

The codebase already seeds **Reference data** ŌĆö the **Currency** rows and the canonical `Opening Balances` **Equity** **Account** ŌĆö through EF Core's `HasData` in `IEntityTypeConfiguration`s, which bakes the rows into the migrations and ships them to every environment. That is exactly right for data the app depends on, but it cannot express the two requirements of **Development sample data**: it must exist *only* in Development, and its **JournalEntry** dates must stay current so time-relative views (month-to-date on the **Dashboard**, the current **Reporting period**) always have data to show. `HasData` values are frozen at `dotnet ef migrations add` time and run unconditionally on every host. So the development ledger is instead built at runtime through EF Core's `UseAsyncSeeding` hook, configured on the options builder in `BalanceDbContext.OnConfiguring` (right after `UseProvider`) and invoked by the existing startup `MigrateAsync`. The callback is guarded by `IHostEnvironment.IsDevelopment()` (which also excludes the `IntegrationTest` and design-time hosts), wipes the previous sample ledger, and rebuilds it with every date re-anchored to `TimeProvider`'s "today". All of the data lives in one place, `Balance.Data/Seeding/` ŌĆö `DevelopmentSeedData` (a `Build(now)` builder holding every row) and `DevelopmentDataSeeder` (the wipe ŌåÆ build ŌåÆ `SaveChanges` orchestration) ŌĆö keeping it out of the production code paths and the `Configurations/` reference seeds.

## Considered options

- **(A, chosen) Runtime `UseAsyncSeeding`, Development-guarded, wipe-and-rebuild each startup.** The EF-suggested mechanism for conditional, computed seed data. It has the runtime clock, so dates can be re-anchored to the current month on every boot ŌĆö month-to-date views are never empty and grow with the real calendar. The wipe is scoped to the domain tables *except* the `HasData` reference seeds (Currencies untouched; the `Opening Balances` **Account** preserved by excluding its fixed `AccountId`) and Identity users, so the app's hard dependencies survive. Cost: any by-hand edits to seeded rows are discarded on restart ŌĆö acceptable for disposable development data.
- **(B, rejected) `HasData` migration seeding (as used for reference data).** Cannot satisfy either requirement: the rows ship to production, and the dates are constants frozen into the migration, so month-to-date views would point at a fixed past month and go empty as the real calendar advances. Correct for **Reference data**, wrong for **Development sample data**.
- **(C, rejected) A custom seeder service invoked beside `MigrateDatabaseAsync` in `Program.cs`.** Would have full DI access, but it abandons the EF-provided hook in favour of bespoke startup plumbing and a second place where "seed the database" lives. `UseAsyncSeeding` gives the same outcome inside the migration call we already make.

## Consequences

- **Seeded rows are disposable.** Every Development startup deletes the prior sample ledger (JournalLines, JournalEntries, BankTransactions + metadata, BankAccounts, Counterparties, and all **Accounts** except `Opening Balances`) and rebuilds it. Hand edits made while developing do not survive a restart; the **Currency** reference rows, the `Opening Balances` **Account**, and Identity users are explicitly excluded from the wipe.
- **The seeder bypasses the service layer**, so its data must be hand-correct against invariants the services normally enforce: every **JournalEntry**'s lines must sum to zero per the **Sign convention**, and any **BankTransaction** seeded as attached must carry `JournalEntryId` plus a bank-side **JournalLine** flipped to `Cleared` whose `Amount` equals the transaction's `Money.Amount` under the deliberate raw-equality match (ADR-0013).
- **A known Development login user** (`dev@balance.local`) is seeded create-if-absent so the seeded shared ledger (ADR-0017) is reachable in the browser. Because the seeder holds only the `DbContext`, it constructs the user by hand ŌĆö `PasswordHasher`, normalized email/username, and security stamp ŌĆö mirroring what `UserManager.CreateAsync` would set. Users are not wiped on re-seed.
- **A few `Reconciled` lines are seeded directly** to exercise the frozen-line display, even though no code path produces that state yet ŌĆö the reconciliation pass is unbuilt (see the glossary's Open Questions). The categorisation flow only ever yields `Cleared`, which is what the bulk of attached sample rows use.
- **No production or test impact.** The hook runs inside the migration call the host already makes; the `IsDevelopment()` guard keeps it out of production, integration tests, and design-time (OpenAPI) hosts.
