# JournalEntry editability is bounded

**JournalEntries** are editable, but the editable surface is narrow by design: entry-level `Date`, `Description`, and `CounterpartyId` may be patched; per-line only `JournalLine.Description` may be patched. `Amount`, `AccountId`, the set of lines (no adds / removes / reorders), and `BankTransactionId` are immutable after creation — corrections to those go through delete-and-recreate. PATCH endpoints accept an RFC 6902 `JsonPatchDocument` (System.Text.Json-based) and model the lines collection as an ID-keyed dictionary (`/lines/{lineId}/description`) so identity travels in the path, "no adds / removes" is enforced by a key-set equality check after apply, and per-line `Id`, `CreatedAt`, and `ReconciliationStatus` survive across edits.

We rejected (a) full editability of every field because the previous wipe-and-recreate logic silently reset `ReconciliationStatus` on every save — marking a line `Cleared` during reconciliation and then editing the entry's description quietly lost the reconciliation state, which is real data loss; (b) strict append-only with reversing entries (the accounting-purist position) because typo-fixing is the common case in personal finance and forcing a reversing entry for every "I meant 'Albert Heijn' not 'Albert Hejin'" correction is hostile ergonomics; (c) keeping `BankTransactionId` editable because once an entry references a BankTransaction that link is part of the import / reconciliation audit trail and rewriting it muddies the trail without a clear win; (d) a positional `JsonPatchDocument<UpdateJournalEntryInput>` over an array of lines because `/lines/0/description` paths are fragile across reorders and don't naturally enforce "no adds / removes" — the dictionary representation puts identity in the path and makes the constraint trivial. The current scope may tighten further to fully append-only with reversing entries; today's surface is the smallest set of edits that supports common corrections without rewriting bookkeeping state.
