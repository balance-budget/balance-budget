# JournalEntry editability is gated per `JournalLine` by `ReconciliationStatus`, exposed as a full-body PUT

**Supersedes [ADR 0005](0005-journal-entry-editability-scope.md).**

A `JournalLine` is editable iff its `ReconciliationStatus == Uncleared`. On an `Uncleared` line, `AccountId`, `Amount`, and `Description` are all editable, the line may be removed, and new lines may be added. On a `Cleared` or `Reconciled` line, `AccountId` and `Amount` are frozen (matched by `Id` against the existing entry) and only `Description` is editable. Entry-level fields keep the ADR 0005 split: `Date`, `Description`, and `CounterpartyId` are editable; `BankTransactionId` is immutable. The wire shape is `PUT /api/journal-entries/{id}` carrying the desired final-state body — the client sends the entry as it should be after the edit, the server assigns `Id` for any line submitted without one, and validates that (a) every line with an `Id` whose current `ReconciliationStatus != Uncleared` appears unchanged in `AccountId` and `Amount`, (b) the line amounts sum to zero per `Currency`, and (c) the standard account / currency / counterparty checks pass. New lines default to `ReconciliationStatus.Uncleared`, mirroring the counter-side default in the **Categorisation flow** (ADR 0014). The PUT does not mutate `ReconciliationStatus` on existing lines — body-supplied status is ignored or validated to match current; flipping status is the job of the (currently unimplemented) reconciliation pass and will get its own endpoint. A cash `JournalEntry` (no `BankTransactionId`, no auto-`Cleared` bank-side line) is therefore fully editable — including the set of lines — while every line remains `Uncleared`; this is the intentional consequence of the per-line gate, and is effectively delete-and-recreate while preserving the entry's `Id`, `CreatedAt`, and any reference to it.

We rejected (a) **keeping ADR 0005's narrow PATCH surface** (only entry-level `Date`/`Description`/`CounterpartyId` plus per-line `Description`) because the common correction case in personal finance is "I miscategorised the counter-side `Account`" and forcing delete-and-recreate for that loses the entry's `Id`, `CreatedAt`, and any external reference for what is bookkeeping-equivalent to changing one field; the original ADR's protection (don't silently reset `ReconciliationStatus`) is recovered by the per-line gate without paying that cost. (b) **Widening the existing PATCH in place** (relaxing the dictionary key-set equality check, allowing RFC 6902 add/remove ops on the lines dictionary) because PATCH's incrementality earns nothing once the server has to validate a final state anyway, new lines have no meaningful client-side key (the server assigns `Id`), and "add/remove allowed for some keys but not others" makes the patch semantics harder to reason about than just "send the desired final state"; PATCH-as-verb-with-PUT-semantics would be misleading. (c) **Two endpoints** — keep ADR 0005's narrow PATCH for tiny field tweaks, add a new PUT for the wider replace — because it gives two paths to the same fields, forces clients to choose, and the narrow PATCH's `JsonPatchDocument` machinery only earns its keep under ADR 0005's deliberately narrow surface. (d) **Gating editability entry-wide on `ReconciliationStatus`** in either direction: "no line is `Reconciled`" lets a `Cleared` bank-side line's `AccountId` be re-pointed, silently rewriting what the bank confirmed; "no line is `Cleared` or `Reconciled`" freezes every imported entry the moment categorisation completes (the bank-side line is auto-`Cleared` per ADR 0014), defeating the motivation that drove this revision. The per-line gate is the only one that protects observed-against-the-bank state without freezing freshly-imported entries. (e) **Allowing `BankTransactionId` to be re-pointed via PUT** because the bank-side line's `AccountId` is structurally tied to `BankTransaction.BankAccount.AccountId`, so re-pointing forces the `Cleared` line's `AccountId` to change, contradicting the frozen-by-`Id` rule; misattributed-to-the-wrong-BT remains a delete-and-recreate operation, consistent with ADR 0005's audit-trail argument which still holds. (f) **Folding `ReconciliationStatus` mutation into the PUT** because combining "edit the bookkeeping shape" with "mark lines reconciled" muddies the operation; the reconciliation pass is a distinct user activity (currently unimplemented) and will get its own endpoint when it lands, at which point the question of whether a manual demotion `Cleared` → `Uncleared` re-unlocks bookkeeping editing becomes load-bearing — today, with no demotion path in the system, the per-line gate is effectively "the counter-side is editable, the bank-side is frozen".
