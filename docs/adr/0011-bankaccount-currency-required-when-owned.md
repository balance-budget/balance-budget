# `BankAccount.CurrencyCode` is required when the BankAccount belongs to an Account

`BankAccount.CurrencyCode` is `NOT NULL` when `BankAccount.AccountId` is set, and nullable when `BankAccount.CounterpartyId` is set. The invariant is enforced as a single-table CHECK constraint alongside the existing XOR over `AccountId` / `CounterpartyId` and the "at least one of `Iban` / `AccountNumber`" rule, and is mirrored in `BankAccountService` validation so user-facing errors carry the specific code rather than surfacing as a raw constraint violation. A migration backfills any existing `AccountId`-tied `BankAccount` rows before the CHECK is applied. A BankAccount you own is the canonical record of a real bank product whose currency must be known unambiguously — every `BankTransaction` posted to it carries a `Money` whose currency must match the BankAccount, and statement imports (ADR 0010) need to reject upfront when the file's currency disagrees with the chosen BankAccount's. A counterparty's BankAccount, by contrast, is metadata used to resolve "the IBAN on the other side of someone else's transaction" — the counterparty's currency may genuinely be unknown to us, and bookkeeping on our side doesn't need it (the `JournalEntry` and its `JournalLines` carry currencies via the `Account`s they touch, not via the counterparty's BankAccount).

We rejected (a) requiring `CurrencyCode` unconditionally on every `BankAccount` because it forces the user to know a counterparty's currency just to record their IBAN, which is friction without bookkeeping value — most counterparty BankAccounts are captured as a side effect of an import resolving "this IBAN appears on the other side" and the system has no reliable way to infer that counterparty's currency at that moment. (b) Leaving `CurrencyCode` always-optional (the status quo before this ADR) because it produces silent data drift: an EUR statement landing into a USD-tagged or null-currency BankAccount would either mix currencies in the register or default into the wrong assumption with no error at import time — the very class of bookkeeping ambiguity the codebase otherwise spends effort to avoid (typed `CurrencyCode`, the strict `Money` value object, per-Account currency inheritance). (c) Inferring `CurrencyCode` from the first imported `BankTransaction` because it is magical and one-shot: the first import (or any manual JournalLine on the linked Account) silently determines the BankAccount's currency forever, with no UX prompt, no way for the user to correct an outlier first row, and no story for BankAccounts that exist before any transaction is recorded. (d) Enforcing the rule only in `BankAccountService` validation and skipping the DB CHECK because defence-in-depth is lost — raw-SQL fixups (data migrations, manual remediation, future seed scripts) can violate the invariant and the violation only surfaces much later, far from its cause; the CHECK costs nothing at write time and catches every path that bypasses the service. (e) Modelling owned vs counterparty BankAccounts as two separate tables to lift the invariant into structure because it is a strictly larger schema change for a single conditional NULL, duplicates identity across two tables, complicates the `BankAccount` registry that imports rely on as a single lookup surface, and breaks the existing convention where the XOR over `AccountId` / `CounterpartyId` is the only partitioning the table needs.
