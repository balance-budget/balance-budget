# BankAccount currency is required when the BankAccount is one of yours

A **BankAccount** must have a non-null `CurrencyCode` whenever `AccountId IS NOT NULL` (i.e. it represents one of your own accounts); when `CounterpartyId IS NOT NULL` (it represents a counterparty-owned account), `CurrencyCode` may stay null. The rule is enforced as a single-table CHECK constraint `("AccountId" IS NULL OR "CurrencyCode" IS NOT NULL)` and is mirrored in `BankAccountService.CreateAsync` / `UpdateAsync` via an `EnsureCurrencyWhenOwned` helper that returns `InvariantError("bank_account.currency_required_when_owned", …)` so the API surfaces a 422 with a stable code instead of a 500 from a constraint violation. The motivation is import safety (issue #50): a statement importer compares the file's implicit currency against the target BankAccount's `CurrencyCode` to refuse a misclick like uploading a EUR statement against a USD account, and that check collapses to a single equality if a configured owned account is guaranteed to carry a currency. Treating currency as required-on-create instead of "set on first import" keeps the BankAccount edit surface explicit — imports do not mutate the BankAccount as a side effect — and avoids a tri-state ("currency is null, will be inferred on first import") that the rest of the system would have to reason about.

We rejected (a) leaving `CurrencyCode` fully optional and asserting the rule only inside the import flow, because the invariant is then enforced in one corner of the codebase, the schema permits broken rows, and any non-import write path (manual JSON Patch, future bulk-edit UI, ad-hoc DB fix) silently leaves the BankAccount unimportable; (b) making `CurrencyCode` *unconditionally* required on every BankAccount, because counterparty-owned BankAccounts often carry just an IBAN harvested from a statement-row counterparty field — the user does not know or care what currency that recipient's bank uses, and forcing a guess for data that is never read is gratuitous friction; (c) auto-setting `CurrencyCode` to a workspace default (e.g. "EUR") at create time when AccountId is set and CurrencyCode is null, because the "default" is the user's first BankAccount's currency, not the system's — there is no globally correct default, and silently picking one would surface as a confusing import rejection later ("why did this USD statement fail against a EUR account I never configured?"); (d) inferring `CurrencyCode` from the first successful import and persisting it as a side effect, because BankAccount edits should be explicit user actions — surprising mutations as a byproduct of an upload make the audit trail worse and the failure modes harder to reason about; (e) modelling currency on the BankAccount as a list of allowed currencies rather than a single value, because the ING current-account format we're targeting is single-currency per account and the multi-currency case is genuinely a separate `BankAccount`, not a per-row currency-tagged feed. The accepted cost is that creating an owned BankAccount now requires one more required field on the form and updating existing tests that previously took `CurrencyCode: null` for owned accounts; both are one-time fixes, and Migration A backfills nothing because current dev data has `CurrencyCode` set on every `AccountId`-tied row (the migration is defensive: it will fail loudly rather than silently substitute a guessed currency if it ever finds a bad row).
