# Multi-user with a single shared ledger

The app supports multiple human logins but exactly one ledger; every authenticated user has identical, full access to every **Account**, **JournalEntry**, **BankAccount**, **Counterparty**, and **BankTransaction**. There is no `OwnerId` / `UserId` / `TenantId` column on any domain entity, no per-user query filter, and no role hierarchy. The intended deployment is a self-hosted household tool used by a couple — "give me and my partner our own logins on the same books" — not a multi-tenant SaaS. Identity exists at the access boundary only: it gates whether a request is admitted at all, never which rows it can see.

We rejected (a) full multi-tenancy with per-user data isolation (`OwnerId` on every aggregate, every query filtered, separate seed data per user), because the user constituency is *one household sharing one set of books* — duplicating the ledger per person would make co-managing finances actively harder, and the cross-cutting `OwnerId` would touch every endpoint, projection, test fixture, and import path for the benefit of nobody who exists; (b) a single-user-with-a-login model (one row in `AspNetUsers`, treated as the system owner), because we genuinely want two humans to log in separately with their own passwords so that revoking one does not lock out the other and so that the audit-shaped questions ("whose session was this?") are at least *answerable* even if not currently *answered*; (c) attribution on domain entities (`CreatedByUserId` / `UpdatedByUserId` on `JournalEntry` and friends), because the social model is a household — when something looks weird you ask the other person verbally — and propagating an `ICurrentUserAccessor` into every service for a question we'd rarely look at is mechanical work without a corresponding payoff (attribution can be added later as an additive, nullable migration; historical rows simply stay null and the UI says "before attribution"); (d) an `Admin` role with the first user as gatekeeper for user management, because the relationship being modelled is a partnership, not a hierarchy — encoding "one of you is the boss" in the schema would misrepresent the actual social structure, and any user-management foot-gun we worried about (rage-disable, accidental self-lockout) is cheaper to prevent with two validation rules than with a whole role system.

The two foot-gun guards on the user-management surface ("you cannot disable yourself", "you cannot disable the last active user") are the explicit price of the flat model: they exist so that the absence of an admin role does not translate into the absence of any safety net. The day either guard becomes insufficient — e.g. the user set grows beyond a household and "anyone can lock out anyone" stops sounding fine — that is the day to revisit this ADR and add `IdentityRole<UserId>`; the change is additive and not blocked by anything decided here. Similarly, if attribution becomes load-bearing later (e.g. a reconciliation pass needs to know who marked a line `Cleared`), `CreatedByUserId` is a nullable column-add against the same Identity user table, not a re-architecture.
