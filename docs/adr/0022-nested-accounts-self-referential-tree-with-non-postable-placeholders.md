# Nested accounts are a self-referential `Account` tree with non-postable placeholder parents

**Accounts** form a tree via a nullable `Account.ParentAccountId` self-reference, to arbitrary depth — the **Chart of accounts** (see [CONTEXT.md](../../CONTEXT.md)). Every Account is either **Postable** (a leaf that **JournalLines** may reference) or a non-postable placeholder (a parent whose balance is the recursive signed sum of its descendant leaves). The purpose is reporting roll-up: "Food" totals "Groceries" + "Dining" without itself being posted to. Parents are not a new entity — they are ordinary `Account` rows with `IsPostable = false`.

Postability is an **explicit boolean** (`Account.IsPostable`), not derived from whether the account has children. Three invariants hold: (a) a non-postable Account is never referenced by a **JournalLine**; (b) an Account with children is never Postable; (c) an empty placeholder (no children yet) and an empty leaf (no lines yet) are both legal. The flag changes only via an explicit conversion — a leaf flips to placeholder only with zero JournalLines, a placeholder flips to leaf only with no children — so nesting a child under a Postable leaf is rejected until the parent is converted first. Re-parenting is allowed and re-rolls balances automatically (they are derived), guarded against cycles. A parent with children cannot be deleted (FK `RESTRICT`).

Every Account in a subtree shares one **AccountType** and one **CurrencyCode** (the homogeneity rule), so the rolled-up balance is a single coherent signed sum in one currency, and the sign convention (ADR-0012) applies unchanged. The currency half of the rule is a write-time constraint we can relax when multi-currency lands; the type half is permanent (mixing debit- and credit-normal accounts under one parent would make the roll-up a net-worth figure, a different and more invasive feature). A parent's **Register** is the union of its descendant leaves' JournalLines, merged newest-first, with no elimination of intra-subtree self-transfers (both legs show; they net to zero in the balance).

Separately, this removes the global `UNIQUE` constraint on `Account.Name` and introduces `Account.Code` — a required, globally-unique short **string** (the chart-of-accounts "account number", e.g. `5110`). `Name` now carries no uniqueness at all; `Code` is the canonical human key. Hierarchical code numbering (a child's code sharing its parent's prefix) is convention, not enforced — only global uniqueness is. The seeded `Opening Balances` equity account is backfilled with code `3900`.

We rejected:

(a) **A separate `AccountGroup` entity** owning flat, always-postable leaf Accounts, instead of a self-referential tree. It would keep the "every Account is postable" invariant pristine, but the roll-up needs a parent to carry `AccountType` and `CurrencyCode` anyway, so `AccountGroup` would duplicate both columns and need rules keeping group-type in sync with member-type — the same constraints spread across two tables — plus a dual hierarchy (a tree of groups *and* a flat set of accounts) for every query and the frontend to merge. A self-referential tree is one recursive structure and matches how GnuCash/Firefly and real charts of accounts model this.

(b) **Deriving postability from child count** (an Account is non-postable iff it has children) instead of an explicit flag. Simpler in that it needs no column, but it means an empty account's nature is undefined until the first child or first line lands, and the parent/leaf transition happens as an implicit side effect of nesting. We chose the explicit flag so postability is a deliberate, inspectable property and nesting never silently mutates it.

(c) **Heterogeneous parents** (mixed `AccountType` under one parent, e.g. a checking Asset and its credit-card Liability under "ABN AMRO"). This forces a typeless parent and turns the roll-up into a net-worth figure spanning debit- and credit-normal accounts — a coherent but much larger feature. Homogeneity keeps the parent a normal typed Account and the roll-up a normal balance.

(d) **Postable parents** (GnuCash-style, where any account can both carry postings and have children, parent balance = own lines + descendants). More flexible, but it complicates the leaf/posting model and every balance query with an "own lines" term that is always zero under our placeholder model. We chose strictly non-postable parents.

(e) **Keeping `Name` globally unique** (or making it unique among siblings). Global uniqueness forces awkward disambiguated names ("Banking Fees" vs "Investment Fees") that worsen as the tree grows; sibling-uniqueness is a usability guard but still a constraint. We dropped name uniqueness entirely in favour of a required, unique `Code` — the standard chart-of-accounts key — leaving `Name` free. `Code` is deliberately *not* named `AccountNumber` to avoid colliding with `BankAccount.AccountNumber` (the bank-side identifier), an ambiguity CONTEXT.md already flags.
