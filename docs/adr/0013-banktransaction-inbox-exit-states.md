# `BankTransaction` inbox exits via Categorise, Attach, or Dismiss; multiple `BankTransaction`s may share a self-transfer `JournalEntry`

A `BankTransaction` leaves the **Inbox** in exactly three ways: (a) **Categorise** — a new `JournalEntry` is created via the **Categorisation flow** with `BankTransaction.JournalEntryId` set to the new entry; (b) **Attach** — `JournalEntryId` is set to an existing `JournalEntry` that represents the same self-transfer reported from the other side, and the matching `JournalLine` is flipped from `Uncleared` to `Cleared`; (c) **Dismiss** — `DismissedAt` (UTC timestamp) and `DismissedReason` (short free-text) are set on the row.

The link between a `BankTransaction` and its `JournalEntry` lives on the `BankTransaction` side as a nullable scalar FK `JournalEntryId`. A `BankTransaction` references at most one `JournalEntry` (enforced by the scalar FK). A `JournalEntry` is referenced by zero, one, or many `BankTransaction`s — the multi-BT case is reserved for self-transfers, where the same money movement is reported by two own-bank statements (e.g. a Current → Savings transfer produces one row on each statement). The service layer enforces "multi-BT only when self-transfer" as part of the Attach validation: a `BankTransaction` may only attach to a `JournalEntry` whose every line is on an `Account` that has a backing `BankAccount` (i.e. one of yours) and whose `CounterpartyId IS NULL`. There is no DB-level CHECK constraint for this — the invariant is unreachable through the API, and the alternative (denormalised `IsSelfTransfer` flag plus a trigger or PG-specific partial constraint) is overengineered for a guard that only one write path can violate.

The **Attach predicate** drives both the Inbox hint surface and the Attach endpoint's server-side validation. A `BankTransaction` BT may attach to a `JournalEntry` JE iff: (1) `BT.JournalEntryId IS NULL AND BT.DismissedAt IS NULL`; (2) there exists `JournalLine L1` on JE where `L1.AccountId == BT.BankAccount.AccountId`, `L1.ReconciliationStatus == Uncleared`, and `L1.Amount` matches `BT.Money.Amount` with signs aligned in the focal-account frame (positive BT = money in to the BankAccount; matches a debit on an `Asset` line and a credit on a `Liability` line); (3) there exists a different `JournalLine L2` on the same JE whose `Account` has a backing `BankAccount` with `Iban == BT.CounterpartyAccountNumber`; (4) `|BT.BookingDate - JE.Date| <= 7 calendar days`; (5) `L1.Account.CurrencyCode == BT.Money.CurrencyCode`; (6) every other line on JE is on an own-`Account` (the self-transfer shape guard); (7) `JE.CounterpartyId IS NULL`.

The **Inbox hint** is computed server-side by the Inbox list endpoint. Each Inbox row carries an optional `MatchingJournalEntryId` plus a small inline summary `{ Id, Date, Description?, OtherAccountName }`. If exactly one `JournalEntry` satisfies the predicate, the hint is populated and the SPA surfaces a one-click `Attach` action on the row alongside the existing `Categorise` / `Dismiss` affordances. If zero or multiple match, no hint is shown — multi-match cases route the user to the manual JE-picker fallback inside the Categorisation flow (ADR 0014), and zero-match cases default to Categorise-as-new. The hint computation is a single LEFT JOIN with conditions all backed by existing indexes (`JournalLine.AccountId + ReconciliationStatus`, `JournalEntry.Date`); detecting at *read* time keeps the import path (ADR 0010) write-side-clean and avoids the "an import silently mutated a JournalEntry" failure mode.

The **endpoint surface** mirrors `dismiss` / `undismiss`: `POST /api/bank-transactions/{id}/attach` (body `{ JournalEntryId }`) and `POST /api/bank-transactions/{id}/detach`. Attach validates the predicate above, sets `BT.JournalEntryId`, flips the matching `JournalLine`'s `ReconciliationStatus` from `Uncleared` to `Cleared`, and returns the updated `JournalEntryOutput`. Detach does the inverse: sets `BT.JournalEntryId = NULL` and flips the previously-attached line back to `Uncleared` (the line's "the bank confirmed it" semantic disappears when the BT does). Re-attaching to a different `JournalEntry` is detach-then-attach across two client calls — no single-atomic-move endpoint, because the window of inconsistency is one client roundtrip and the alternative invents a third verb for a rare case. Neither Attach nor Detach goes through `PUT /api/journal-entries/{id}` (ADR 0016); they are state transitions on the BT↔JE link, not bookkeeping edits, so they don't interact with the per-line editability gate. The 2-`Cleared`-line case the per-line gate naturally produces for an attached self-transfer JE is exactly the right behaviour — both bank-side lines are frozen in `AccountId` / `Amount` because both have been observed against a bank.

**Dismiss** stays for the cases that have no `JournalEntry` to point at: test transactions, fee corrections booked elsewhere, rows the user explicitly chooses not to categorise. `DismissedAt` and `DismissedReason` are set and cleared through the existing `POST /api/bank-transactions/{id}/dismiss` and `/undismiss`; the dismissal carve-out from ADR 0010's "BankTransaction is immutable" applies to bank-supplied fields only and is unchanged.

We rejected (a) **keeping the 1:1 cardinality and routing every sibling-of-self-transfer BT through Dismiss** because the user-facing decision on the second-side BT was "you have to know this is a sibling, look up the matching JE, and write a free-text reason explaining the link" — both steps the system should automate — and the resulting Dismissed rows are structurally lying about what happened (the row *was* a real money movement, just one we already booked from the other side). The link is what the data wants to express; Dismiss-as-sibling was the workaround for not having it. (b) **An explicit line↔BT join table** (`(JournalLineId, BankTransactionId)`) over the implicit `AccountId`-derived mapping because every `JournalEntry`'s lines are on distinct `Account`s, every `Account` has at most one `BankAccount`, and every `BankTransaction` belongs to one `BankAccount` — so the BT-to-line mapping is unique by construction without storing it; the join table would only earn its keep if a JE could have two `Cleared` lines on the *same* `Account`, which is bookkeeping nonsense for a self-transfer. (c) **Keeping the FK on the `JournalEntry` side** (`JournalEntry.BankTransactionId?`) and adding a separate "secondary BTs" mechanism, because two ways of expressing the same relationship in one schema is more cardinality machinery than the problem needs; flipping the FK to the BT side gives N:1 directly with one nullable column and one mapping query. (d) **Auto-attach silently at import time** (the matching predicate fires inside the import service and sets `JournalEntryId` without user confirmation) because it violates ADR 0010's "imports only write `BankTransaction`s; categorisation is a separate flow" boundary, mutates `JournalEntry` state (an `Uncleared` line silently becomes `Cleared`) as a side-effect of a CSV upload, and bites once when the match is wrong — different transfer, same amount, same week — at which point the user discovers a `JournalEntry` was altered by an import they didn't realise had categorisation power. (e) **Fuzzy or name-based matching in the predicate** (looser date window, amount tolerance for FX rounding, counterparty-name similarity) for the same reason ADR 0014 rejected fuzzy counterparty matching at flow open: silent wrong-attach is the worst failure mode, and the manual JE-picker fallback covers near-misses without polluting the auto-hint with false positives. (f) **A DB-level CHECK constraint** enforcing "JE with multiple BTs must be self-transfer" because the natural expression requires denormalising `AccountType` onto `JournalEntry` or relying on a trigger / PG-specific partial index — both heavier than a single service-layer query that runs inside the Attach transaction, and the invariant is unreachable through the API surface anyway. (g) **A `move` endpoint for re-attaching a BT to a different JE in one atomic call** because re-attach is rare, the intermediate-detach window is one client call wide, and the third verb obscures that re-attach is just two existing operations composed. (h) **A sentinel `Ignored` Account that consumes dismissed rows via a one-line clearing JE** because it pollutes the ledger with bookkeeping noise that has nothing to do with money movements, makes reports lie ("you spent €0 on Ignored last month" is not a useful row), and breaks the invariant that `JournalEntries` represent real bookkeeping events. (i) **A separate `BankTransactionDismissal` side table** to preserve strict per-column immutability of `BankTransaction` because the audit benefit is negligible in a single-user system (dismissals are not facts about money, only about user intent), the side table doubles the write cost for the dismiss action without buying anything queryable, and the conceptual "immutability" we care about is the bank-supplied facts — placing user metadata on the same row is honest about who applied it and when, and keeps the inbox-filter query a single-table predicate. (j) **Hard-deleting `BankTransaction` as the "make it stop showing" action** because it forfeits idempotent re-imports (ADR 0010): the row would re-appear on the next CSV import with the same `RowHash` and re-enter the **Inbox**, undoing the user's decision; **Dismissed** is the correct soft state — the `RowHash` stays in the unique-index dedup, and the dismissal survives re-import.
