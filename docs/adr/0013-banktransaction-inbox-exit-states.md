# `BankTransaction` inbox exits via `JournalEntry` creation OR explicit dismissal; at most one `JournalEntry` per `BankTransaction`

A `BankTransaction` leaves the **Inbox** in exactly two ways: a `JournalEntry` is created referencing it via `BankTransactionId`, or the row is **Dismissed** — a user action that sets `DismissedAt` (UTC timestamp) and `DismissedReason` (short free-text) on the `BankTransaction` itself. The 1:1 cardinality between `BankTransaction` and `JournalEntry` is enforced by a filtered `UNIQUE` index on `JournalEntries.BankTransactionId WHERE BankTransactionId IS NOT NULL`; multi-account splits are modelled as multiple `JournalLines` within a single `JournalEntry`, never as multiple `JournalEntries` sharing one `BankTransaction`. The dismissal carve-out from ADR 0010's "BankTransaction is immutable" applies to bank-supplied fields only (`BankAccountId`, `BookingDate`, `Money`, `Description`, `CounterpartyName`, `CounterpartyAccountNumber`, `RawSource`, `RowHash`); `DismissedAt` and `DismissedReason` are user-applied metadata recorded on the row, set and cleared through dedicated endpoints (`POST /api/bank-transactions/{id}/dismiss` and `POST /api/bank-transactions/{id}/undismiss`) — never editable via PATCH, never set as a side effect of the **Categorisation flow** (ADR 0014), and never affecting `RowHash`-based deduplication on re-import.

We rejected (a) **inbox-exits-only-via-`JournalEntry`-creation** because the deferred self-transfer case (ADR 0014: v1 does not auto-detect the second side of a between-own-accounts transfer) leaves the orphaned sibling sitting in the **Inbox** forever with no way to acknowledge "no JE needed here", and a long tail of other "should not become a JE" rows — test transactions, fee corrections booked elsewhere, rows the user explicitly chooses to ignore — hit the same wall, accumulating permanent clutter that the user cannot clear without polluting the ledger. (b) **A sentinel `Ignored` Account that consumes dismissed rows via a one-line clearing JE** because it pollutes the ledger with bookkeeping noise that has nothing to do with money movements, makes reports lie ("you spent €0 on Ignored last month" is not a useful row), and breaks the invariant that `JournalEntries` represent real bookkeeping events. (c) **A separate `BankTransactionDismissal` side table** to preserve strict per-column immutability of `BankTransaction` because the audit benefit is negligible in a single-user system (dismissals are not facts about money, only about user intent), the side table doubles the write cost for the dismiss action without buying anything queryable, and the conceptual "immutability" we care about is the bank-supplied facts — placing user metadata on the same row is honest about who applied it and when, and keeps the inbox-filter query a single-table predicate. (d) **Allowing many `JournalEntries` per `BankTransaction`** (i.e. not adding the filtered `UNIQUE` index) because the only motivating case — splitting one deposit across multiple invoices into multiple JEs — composes wrong with double-entry: each "partial JE" would have to fabricate its share of the asset-side line, double-counting the bank movement; multi-line JEs are the natural and only correct shape for splits, and the `UNIQUE` index turns the alternative into a database error rather than a silent footgun. (e) **Hard-deleting `BankTransaction` as the "make it stop showing" action** because it forfeits idempotent re-imports (ADR 0010): the row would re-appear on the next CSV import with the same `RowHash` and re-enter the **Inbox**, undoing the user's decision; **Dismissed** is the correct soft state — the `RowHash` stays in the unique-index dedup, and the dismissal survives re-import.
