# Typed IDs over UUIDv7

Every entity's primary key is a strongly-typed `readonly record struct` wrapper over a UUIDv7 `Guid` — e.g. `record struct AccountId(Guid Value)`, `record struct JournalEntryId(Guid Value)`. IDs are generated client-side via `Guid.CreateVersion7()` (time-sortable, so clustered indexes stay happy on Postgres). The structs, their EF Core value converters, and their `System.Text.Json` converters are emitted by a source generator (e.g. `StronglyTypedId`) to keep boilerplate manageable. `BaseEntity` becomes generic: `BaseEntity<TId>`. `Currency` keeps its natural string key, wrapped as `record struct CurrencyCode(string Value)` for the same type-safety benefit.

We rejected (a) the existing `int` identity columns because mixing an accidentally-swapped `AccountId` and `CounterpartyId` in cross-table FKs is a real bug class, sequential IDs leak scale through public URLs, and they foreclose any future device-sync / multi-user migration; (b) raw `Guid` without typed wrappers because it solves identity but not parameter-swap safety; (c) typed IDs over `int` because it preserves the sync-migration cost. UUIDv7 specifically is preferred over UUIDv4 because random GUIDs make poor clustered-index keys on Postgres, while v7's time-sortable prefix preserves insert locality at no semantic cost.
