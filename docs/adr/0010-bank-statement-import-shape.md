# Bank-statement import records `BankTransaction`s only; dedup is per-`BankAccount` SHA-256 of the raw row

Importing a bank statement consumes a CSV stream against a user-chosen `BankAccountId`, validates that the file's `Account` column matches the chosen `BankAccount.Iban`/`AccountNumber` and that the BankAccount's currency matches what the format produces, extracts one `BankTransaction` per statement row, and persists only the rows whose `RowHash` is not already on that BankAccount. No `JournalEntry` is auto-created — categorisation is a separate later flow. `BankTransaction` gains denormalised universal fields `Description`, `CounterpartyName?`, `CounterpartyAccountNumber?` (every statement row carries these regardless of bank), plus `RawSource` (the original row text) and `RowHash` (SHA-256 hex of the raw row bytes after `\r\n → \n` and trailing-whitespace normalisation). Bank-specific extras (ING's `TransactionCode`, SEPA mandate ID, foreign-currency block) stay inside `RawSource` and are not promoted to columns. Uniqueness is `(BankAccountId, RowHash)` — identical-looking fee rows on two separate accounts of yours remain distinct, while the unique index acts as the durable backstop behind in-memory dedup. Architecture is `IBankTransactionExtractor` per bank (parses, validates per-bank invariants, maps to unsaved `BankTransaction`s) plus a bank-agnostic `IBankTransactionImportService` that loads the BankAccount, calls the extractor, queries existing `RowHash`es in one batched read, partitions extracted rows into new vs duplicate in memory, bulk-inserts the new ones in a single DB transaction, and retries once on a unique-index conflict to absorb the concurrent-import race. The result DTO is `(int Imported, int SkippedAsDuplicate)`; malformed rows fail-fast and roll back the whole transaction. Upload is `POST /api/bank-accounts/{id}/imports` as `multipart/form-data`. v1 ships ING current-account only with a single extractor registered — dispatcher / `StatementFormat` enum is deferred until format #2 lands.

We rejected (a) auto-creating `JournalEntry`s at import time (the default for most personal-finance tools) because it would pollute the ledger with low-confidence postings tied to delete-and-recreate ergonomics (ADR 0005) and conflate "what the bank told us" with "what we did about it" — the boundary ADR 0003 was explicitly drawn to keep separate; categorisation belongs in a later UI flow that consumes unmatched `BankTransaction`s. (b) Hashing parsed fields instead of raw row bytes because any change to the parser would silently change historical hashes and break re-import idempotency; raw bytes make the hash a property of what the bank said, independent of parser version. (c) Including `BankAccountId` in the hash input because it conflates row content with where the row is filed and breaks if a BankAccount FK is ever fixed up — per-row uniqueness scoping belongs in the index, not the hash. (d) Globally-unique `RowHash` because identical-looking €0.10 fee rows on two separate ING accounts of yours are real, distinct events that should not collide. (e) Storing only `RawSource` + `RowHash` and reparsing on every list query because the upcoming "unmatched BankTransactions" view needs `Description` and counterparty fields available cheaply; the three universal fields earn their column status because every statement row across every bank carries them (CAMT.053, MT940, generic CSV). (f) Per-bank tables (e.g. `IngBankTransaction` joining `BankTransaction`) because the only field savings are bank-specific extras that can live inside the `RawSource` blob, while joins would cost on every read and the schema would sprawl per supported bank. (g) Best-effort partial-success on malformed rows because it composes badly with content-hash dedup (failed rows have no stored hash, so the next import re-encounters them and the user has to remember which rows weren't imported) and ING CSVs are machine-generated — malformed rows are practically nonexistent. (h) An `Import` audit-trail entity with `ImportId` FK on `BankTransaction` because no v1 UI consumes it, Q7's strict IBAN match nearly eliminates the "undo a wrong import" use case it would enable, and idempotent re-imports make per-import counts murky (the second import has `ImportedCount=0, SkippedAsDuplicateCount=N` — what does "delete this import" mean for it?); a nullable FK can be added in a future migration without touching existing data. (i) Per-bank-format endpoint paths (`POST .../imports/ing-current-account`) because they proliferate routes per bank and fragment OpenAPI without a clear benefit over a single endpoint with a future `StatementFormat` selector. (j) Building the dispatcher / keyed-DI / format-enum machinery now for forward compatibility because it is the textbook premature abstraction with one registered impl, and the refactor when format #2 lands is small and surgical — swap one `AddScoped` for keyed-DI, add an enum field, add the dispatcher. (k) JSON body with the CSV base64- or text-encoded inside because it inflates payload, requires the SPA to read-then-encode the file before posting, and forfeits the `IFormFile` framework support (size limits, content-type sniffing, filename metadata) that multipart gives for free. (l) Serializable transactions or per-BankAccount application locks for the concurrent-import race because the unique index already provides the safety net and a single in-process retry covers the only realistic race (double-clicked submit button); locks would need persistence and distributed coordination for negligible benefit.
