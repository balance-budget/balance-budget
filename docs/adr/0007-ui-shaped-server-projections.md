# Account list and Register are UI-shaped server projections

`GET /api/accounts` returns each `AccountOutput` enriched with the account's current `Balance` (a `Money` summed over its `JournalLines`) and a nullable `BankAccount` summary (`Iban?`, `AccountNumber?`, `Bic?`, `BankName?`) from the optionally-linked **BankAccount**; the single-account `GET /api/accounts/{id}/balance` stays for balance-only callers. `GET /api/accounts/{id}/register?skip&take` projects the focal account's **JournalLines** into **RegisterRow**s server-side. Each row carries the focal-account-signed `Money` amount (positive = money in, negative = out — *not* the raw debit/credit sign), the **JournalEntry** header (`Date`, `Description`, `CounterpartyId`/`CounterpartyName`), the focal line's `ReconciliationStatus` and `Description`, and the offsetting side as a list of `(AccountId, AccountName, Amount)` — one element for a two-leg entry, multiple for a split. The list orders `Date DESC, JournalEntryId DESC` and pages with `skip`/`take` like `ListJournalEntries`. Frontend `api/*.ts` mappers brand IDs and build view-models directly — no client-side joining, sign-flipping, or split-handling.

The projection lives server-side because three concerns — the focal-account sign flip (debit-normal vs credit-normal **Accounts** show the same line with opposite UI signs), the split shape (one row per focal line), and offsetting-account name enrichment — would otherwise be repeated by every TS consumer. Splits are kept as one focal row (not exploded) so the focal-amount column matches what a bank statement shows for reconciliation against a **BankTransaction**. The accepted trade-off: `AccountOutput` is no longer a thin entity mirror and **RegisterRow** is a view-only shape that never round-trips back — preferable to the "endpoint = entity" purity that `lib/domain.ts` drift already disproved.
