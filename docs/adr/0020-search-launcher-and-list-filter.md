# Two distinct search surfaces: a Cmd-K launcher for navigation and per-list `?q=` filters for in-list narrowing

Search arrives as two deliberately separate surfaces rather than one unified search experience. (1) A **launcher** invoked by Cmd-K (or by clicking the topbar input, which now opens a modal instead of accepting keystrokes) takes a query against a single backend endpoint `GET /api/search?q=...&take=5` and returns grouped, slim per-type results — **Account** by `Name`, **Counterparty** by `Name`, **BankAccount** by `Iban`/`AccountNumber`/`CardIdentifier`/`BankName`/`AccountHolderName` (with the query stripped of spaces and uppercased when matched against `CardIdentifier` so the stored normalised form round-trips), and **JournalEntry** by `Description` — plus an in-memory match against static page labels. Each section caps at 5 with a `+ N more matching` footer when truncated, sorted alphabetically by name except JournalEntry which sorts newest-first by `Date`; the result envelope carries a per-section `TotalCount`. Picking any result navigates to that entity's detail page; **BankTransaction** is intentionally not a launcher entity in v1 because a categorised BT is best reached via its referencing **JournalEntry** and an uncategorised BT belongs to the Inbox workflow, not a navigator. (2) A **list filter** is exposed as a `?q=` query string on the existing list endpoints for **BankTransaction** (matching `Description + CounterpartyName`) and **JournalEntry** (matching `Description` only); the filter composes with the existing `Inbox/Matched/Dismissed/All` BankTransaction filter, persists in the URL alongside `page` and `filter`, resets `page` to 1 on every query change, and applies plain `field.Contains(term)` — the SQLite-vs-Postgres case-sensitivity difference is accepted because a deployment runs one provider or the other, never mixed. Both surfaces use a 200 ms debounce; the launcher additionally requires a 2-character minimum (the list filter has none — every character narrows something already visible). To make a real page-number paginator possible, every list endpoint's response is uniformly rewrapped from `IReadOnlyList<T>` to `PagedOutput<T> { Items, TotalCount }` and the paginator renders a sliding window with first/last anchors (`‹ 1 … 21 22 [23] 24 25 … 47 ›`, all pages when total ≤ ~7) plus the existing `from–to of total` range text; `PAGE_SIZE = 50` stays constant with no user-facing selector.

We rejected (a) **a single unified search results page** that merges launcher and list-filter into one experience, because the load-bearing use case for the list filter is *bulk-categorising every Albert Heijn row in the Inbox* — which requires filter-on-visible-list semantics with the selection model intact across pagination, not a flat results page; and because cross-type relevance ranking is a tuning problem we do not want to own at v1 (every wrong ordering becomes a complaint). (b) **A "launcher opens the list screen with `q` prefilled" model** that would have made the two surfaces share an implementation, because it breaks the cross-type case (a query matching a Counterparty *and* a JournalEntry description has no single list to land on) and forces the navigation product to inherit the filter product's pagination/selection state. (c) **A medium or wide field scope** for the BankTransaction list filter (adding `Reference`, `MandateId`, `SepaCreditorId`, `RawSource`, `BankTransactionMetadata.StringValue`) because matches against `RawSource` and the metadata blob surface rows the user cannot see matched on, the metadata side-table is intentionally rebuildable from `RawSource` (see CONTEXT.md and the metadata ADR) and is not a stable search surface, and the bulk-categorise use case is satisfied by `Description + CounterpartyName`; the escape hatch for "narrow isn't enough" is to widen later, which is cheap, whereas removing a field once users rely on it breaks expectations. (d) **Query syntax in the search box** (`amount:>50 date:2024-12 albert heijn`) because every system that grows one ends up explaining it to users in a tooltip, parser ambiguity creates surprises (does `2024` match a date or a description?), and the workflows that would justify amount/date filtering (auditing, reconciliation) are not the workflows we are shipping search for; dedicated filter controls — when amount/date earn their keep — are discoverable and unambiguous. (e) **Per-`BankTransaction` results in the launcher**, because there is no per-BT detail screen today (a BT lives inside Inbox/Matched/Dismissed/All), a categorised BT is reachable via the JournalEntry it references, an uncategorised BT belongs to the Inbox workflow not a navigator, and the user's mental retrieval cue for a categorised payment is the JournalEntry description or the Counterparty — not the bank-row description. (f) **Aggregate "view N matching" rows in the launcher** (e.g. `View 47 BankTransactions matching 'albert' in Inbox` deep-linking to `/bank-transactions?q=albert&filter=Inbox`) for v1, because the launcher's first-pass scope is "jump to a named thing"; aggregate rows are easy to add later (they consume the same `TotalCount` per section we already return) and the `q` deep-link target already exists from day one. (g) [**Partially superseded 2026-05-30 — see the Amendment below. The _list-filter_ half is now adopted; only the _launcher_ half of this rejection stands.**] **Searching JournalEntry by its linked `Counterparty.Name`** in both launcher and list filter, because the launcher already returns the Counterparty as its own grouped result, and merging "JEs whose Counterparty matched" into the JournalEntry section bloats it with rows reachable via the Counterparty in one click while conflating two distinct retrieval intents; the same holds for searching Counterparty by linked BankAccount IBAN — an IBAN search returns the BankAccount which links to its Counterparty. This recommendation leans on a Counterparty detail view that lists its JournalEntries, which does not yet exist and is flagged as a follow-up; until it lands, "every JE with Counterparty X" requires the user to fall back to the JE list `?q=` filter. (h) **Reusing each list endpoint for the launcher** (the SPA fires four parallel calls in a `useQueries` hook) instead of a single `/api/search`, because the launcher renders slim rows (id + name + maybe date) and the list endpoints return full per-entity DTOs (balances, currency metadata, full BT field set) which is honest bytes-on-the-wire waste; because `/api/search` returns a per-section cap of 5 by construction, while reusing list endpoints conflates the cap-5 launcher case with the page-50 filter case; and because a single round-trip on every keystroke beats four cancellations and four sets of headers. (i) **A separate `/count` endpoint or an `X-Total-Count` response header** for pagination totals, in favour of folding the count into a `PagedOutput<T>` envelope across every list endpoint uniformly, because the two-call variant risks count/list drift if filters disagree across requests and the header variant is poorly modelled by `openapi-typescript`'s generated client; a one-time uniform envelope change is cheaper than living with two patterns. (j) **Postgres-specific `ILIKE` / functional `LOWER()` index / trigram / FTS** for v1, because at the realistic per-user volumes (10k–100k BankTransactions, hundreds-to-low-thousands of JournalEntries) plain `field.Contains(term)` is milliseconds on either provider, the cross-provider case-sensitivity difference is acceptable for a single-provider deployment, and any of those Postgres features is a one-migration upgrade isolated to `Balance.Data.PostgreSql` when a real performance problem materialises. (k) **A recent-items or command-palette empty state** for the launcher, because recent-items requires hand-rolled tracking (a global nav hook, localStorage shape, eviction) for marginal v1 benefit and the value proposition ("I forgot what I was doing, remind me") is better served by the Dashboard, while command items overlap with the sidebar and tend to grow into a full action surface that is its own product; the empty state is intentionally a placeholder hint so the launcher's purpose stays "jump to a named thing". (l) **Adding `?q=` to the Counterparties, Accounts, BankAccounts, and BankImports list screens**, because the launcher already covers the first three (and the BankAccount/Account picker UIs in the categorisation flow have their own combobox search), the realistic row counts are too small to justify the toolbar real estate, and BankImports is a chronological feed where sort beats search; each can earn `?q=` later when a concrete pain point emerges. (m) **A compact "current/total + jump-to-page" paginator** instead of a sliding-window numbered paginator, because the explicit ask was for visible page numbers and the sliding-window shape (`1 … current ±2 … last`) matches the mental model of every paginated UI on the web; the jump-to-page input is a typing interaction where a click should do. (n) **A user-facing page-size selector**, because `PAGE_SIZE = 50` is already a working default, a selector is a control that changes the URL shape (`?size=…`) and the bulk-select model (selection identity per page) for the marginal benefit of users who want 25-or-100, and adding it later is non-breaking. (o) **Per-service `SearchAsync` methods orchestrated by a thin `SearchService`** (each of `IAccountService`, `ICounterpartyService`, `IBankAccountService`, `IJournalEntryService` gains a `SearchAsync` returning slim hits), because the launcher is *one* user-facing feature with *one* backend endpoint and distributing its implementation across four services to satisfy a layering instinct creates a feature whose logic the next reader has to reassemble from four files; the slim search DTOs (`AccountHit`, `CounterpartyHit`, `BankAccountHit`, `JournalEntryHit`) are search-specific shapes that don't belong on per-entity service surfaces; and the list-screen `?q=` filter is *not* a search-service concern — it lives inline in `BankTransactionService.ListAsync` and `JournalEntryService.ListAsync` as an additional optional parameter, exactly as the existing `BankTransactionListFilter` enum does. (p) **`Task.WhenAll` across the four entity reads inside `SearchService`** on its injected `BalanceDbContext`, because EF Core's `DbContext` is not thread-safe and concurrent operations throw `InvalidOperationException: A second operation was started on this context instance before a previous operation completed`; the safe parallel route exists (`IDbContextFactory<BalanceDbContext>` is already registered in the codebase and could mint a per-task context, each disposed at the end of its task), but at v1 data sizes each per-entity query is sub-10ms, the 200 ms debounce dominates the user-facing latency, sequential `await`s keep `SearchService` a normal context-injecting service consistent with every other service in the codebase, and the parallel-via-factory swap is a localised refactor inside `SearchService` with no contract change when measurement ever demands it. (q) **Touching the existing `MatchingJournalEntry` hint N+1 in `BankTransactionService.ListAsync` (lines 92-97) as part of this work**, because the N+1 is bounded by `PAGE_SIZE = 50` and pre-existed this PR, the 7-condition Attach predicate (ADR 0013) makes a batched re-expression a focused refactor in its own right, and `?q=` *narrows* the result set so search makes the N+1 milder rather than worse; the proper batching fix earns its own PR when it earns one.

## Amendment (2026-05-30): the list filter now matches Counterparty name

Item (g) above rejected counterparty-name matching in **both** the launcher and the list filter. The **list-filter half is reversed.** The `?q=` filter on the **JournalEntry** list (the _Activity_ feed) and on the per-**Account** **Register** now match the entry `Description` **OR** the linked **Counterparty**'s `Name` (case-insensitive, via `DbFunction.CaseInsensitiveLike`).

Why this changed:

- The reservation in (g) leaned on "a Counterparty detail view that lists its JournalEntries, which does not yet exist" — it exists now (the Counterparties top-level nav with a per-counterparty entries view), so the original blocker is gone.
- It aligns the JournalEntry list filter with the **BankTransaction** list filter, which already matches `Description + CounterpartyName`.
- On both surfaces the row's primary heading is `CounterpartyName ?? Description`, so the visible label is now searchable rather than mysteriously unmatchable.

What still stands from (g): the **launcher** half. `/api/search` continues to return **Counterparty** as its own grouped section and does **not** fold counterparty-matched JournalEntries into the launcher's JournalEntry section; searching a Counterparty by a linked BankAccount IBAN is likewise unchanged.
