# Authentication: cookie for SPA, opaque per-user PATs for third parties

ASP.NET Identity Core is the user store and password authority; the SPA authenticates with a same-origin cookie, and third-party callers authenticate with opaque per-user **Personal Access Tokens** of the shape `bal_pat_<32-byte-base64url>`. A default policy scheme sniffs the `Authorization` header — `Bearer bal_pat_` routes to the API-token handler, otherwise to the cookie handler — and a single `RequireAuthorization()` on the `/api` group covers everything, with `AllowAnonymous` carve-outs for the setup wizard, login, healthz, OpenAPI, and Scalar. CSRF uses the canonical ASP.NET antiforgery pattern (HttpOnly cookie token + `X-XSRF-TOKEN` request-token header cached in JS memory), skipped for API-token requests.

Identity is wired as `AddIdentityCore<TUser>().AddSignInManager().AddEntityFrameworkStores<BalanceDbContext>()` — not `AddIdentity` or `AddDefaultIdentity`. `RoleManager` is not registered: there are no roles (ADR-0015). The cookie is `HttpOnly`, `Secure` in production, `SameSite=Strict`, sliding 30-day lifetime; `SecurityStampValidationInterval` is ~5 minutes so user disable (`LockoutEnd`) takes effect on existing sessions within minutes. The password policy is length-first (minimum 12, no composition rules) per NIST SP 800-63B. The API-token handler SHA-256-hashes the random portion and looks it up against `ApiToken.TokenHash`; the row stores `Prefix` and `Last4` but no recoverable secret, and rejects tokens whose owner has a future `LockoutEnd`.

We rejected `MapIdentityApi<TUser>()` and `AddDefaultIdentity` (public `/register` and Razor UI we never render), JWTs for PATs (no instant revocation without a deny-list that defeats the no-DB-hit benefit, and no natural place to name a token), password KDFs for the token hash (PATs carry 256 bits of entropy, so a fast hash matches GitHub's approach), per-endpoint multi-scheme attributes, and `SameSite=Lax` (the email-deep-link argument does not apply to a self-hosted app).

Provisioning is two-stage: a one-shot anonymous `POST /api/auth/setup` gated on **both** `AspNetUsers` being empty **and** a request-supplied token matching `Auth:SetupToken`; it returns `404` (not `409`) once a user exists. Subsequent users come through the authenticated `/api/admin/users/*` surface. There is no `/api/auth/register`. The PAT minting endpoint returns the plaintext token once; rows carry optional `ExpiresAt` (null = no expiry) and `RevokedAt` for revoke-without-delete.
