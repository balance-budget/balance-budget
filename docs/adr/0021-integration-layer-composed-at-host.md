# Bank-statement integrations live in a separate `Balance.Integration.*` layer, composed at the host beside `Balance.Services`

Bank-specific statement importers (today `Balance.Integration.Ing`) live in their own project that **references `Balance.Services`** and implements the Services-owned contract `IBankTransactionExtractor` (`Balance.Services/Contracts/IBankTransactionExtractor.cs`). The host composes it with a third top-level call — `builder.Services.AddBalanceIntegrationIng()` in `Program.cs`, alongside `AddBalanceServices` and `AddBalanceWeb` — rather than nesting it under `AddBalanceServices`. This is a deliberate exception to the "entry points compose at most two layers" rule in [docs/conventions.md](../conventions.md): the integration layer sits *beside* Services, not under it.

The dependency direction forces the shape. The categorisation/import flow (`BankTransactionImportService`) consumes `IEnumerable<IBankTransactionExtractor>` and dispatches by `ImporterKey`/`SupportedType`; the contract therefore belongs to Services. A concrete extractor (`IngBankTransactionExtractor` etc.) needs Services' contracts and domain types (`Money`, `BankTransaction`, `CurrencyCode`), so the integration project must reference Services. If `Balance.Services` were to call `AddBalanceIntegrationIng` to register the extractors, Services would have to reference `Balance.Integration.Ing` — a reference cycle (Services → Integration → Services). The host is the only place that can legitimately see both, so the host is where the registration lives.

Keeping integrations in their own layer (rather than inside `Balance.Services`) means a new bank is a new `Balance.Integration.<Bank>` project plus one `AddBalanceIntegration<Bank>()` line at the host, with no edit to Services. Each integration owns its parsers, CSV row models, and the extractor implementations behind `internal` types; only its `ServiceCollectionExtensions.AddBalanceIntegration<Bank>` is `public`. Services stays bank-agnostic and depends only on the `IBankTransactionExtractor` abstraction.

We rejected (a) **folding the extractors into `Balance.Services`** because it couples the provider-agnostic domain layer to ING's CSV layouts, transaction-code tables, and note-parsing quirks — every new bank would grow Services and its test surface, and the "Services knows nothing about any specific bank" boundary (the same instinct behind the provider-agnostic `BalanceDbContext`, ADR-0004-era reasoning) would erode. (b) **Inverting the contract so `IBankTransactionExtractor` lives in the integration layer and Services depends on integrations** because then Services couldn't dispatch over a stable abstraction without referencing every bank project, which is the cycle above with the arrows flipped. (c) **A separate `Balance.Integration.Abstractions` project holding just the contract**, with both Services and the bank projects referencing it, because the contract is intrinsically a Services concept (it produces `BankTransaction`s for the import flow Services owns) — a standalone abstractions assembly would be one more project to justify for a single interface, and the current "contract lives with its consumer, implementation lives beside it, host wires them" shape is the smaller thing that works. (d) **Registering extractors via assembly scanning at the host** instead of an explicit `AddBalanceIntegrationIng()` extension because the per-layer `AddBalance*` extension is the codebase-wide composition convention; an explicit extension keeps the registration discoverable and consistent with every other layer.
