# Nested accounts: a self-referential tree with non-postable placeholders

**Accounts** form a tree to arbitrary depth via a nullable `Account.ParentAccountId` self-reference — the **Chart of accounts**. Every Account is either **Postable** (a leaf that **JournalLines** may reference) or a non-postable placeholder parent whose balance is the recursive signed sum of its descendant leaves. Parents are ordinary `Account` rows with `IsPostable = false`, not a separate entity.

Postability is an explicit boolean, with three invariants: a non-postable Account is never referenced by a JournalLine; an Account with children is never Postable; empty placeholders and empty leaves are both legal. The flag flips only via explicit conversion — a leaf to placeholder only with zero JournalLines, a placeholder to leaf only with no children — so nesting a child under a Postable leaf is rejected until the parent is converted. Re-parenting is allowed, re-rolls balances, and is rejected on cycles; a parent with children cannot be deleted (FK `RESTRICT`). Every Account in a subtree shares one **AccountType** and one **CurrencyCode** (homogeneity); the sign convention (ADR-0011) applies unchanged. The type half is permanent (mixing debit- and credit-normal accounts would make the roll-up a net-worth figure); the currency half relaxes when multi-currency lands.

Separately, `Account.Name` loses all uniqueness and `Account.Code` — a required, globally-unique short string (e.g. `5110`) — becomes the canonical key. Hierarchical code numbering is convention, not enforced. We rejected a separate `AccountGroup` entity (duplicates type/currency across two tables), deriving postability from child count (leaves an empty account's nature undefined), heterogeneous parents, and keeping `Name` unique (forces awkward disambiguated names).
