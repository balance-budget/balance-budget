# Forward-looking finance is a computed Projection over templates, not posted future entries

The forward-looking feature (the **Outlook** section — expected recurring payments, "savings after a year", "what's left in checking each month") is a thin derived layer *over* the ledger, never a parallel set of speculative postings. It reuses the loan **Projection** model (ADR-0025), generalized: a **Projection** is *computed, never stored*, anchored to *(balances now, horizon)*, with past periods coming from ledger actuals and future periods from the computation. The loan amortization curve and the liquid-balance curve are two flavors of one concept.

The only new *stored* concept is the **JournalEntryTemplate**: a user-confirmed, `JournalEntry`-*shaped* pattern pinned to exactly one **Liquid** balance-sheet Account, carrying a **Cadence** (`Once` / `Weekly` / `Monthly` / `Quarterly` / `Yearly`), an expected signed `Money` amount, an optional `Counterparty`, and a matching key. It is never posted and owns no `JournalLines`; a `Once` cadence captures a planned one-off, every other cadence a recurring item. Templates are *hybrid-sourced* — the canonical entity is always user-confirmed, but the system proposes candidates mined from history (an Inbox-style accept/dismiss queue), keyed primarily on the SEPA `MandateId` / `SepaCreditorId` already promoted onto `BankTransaction`. Expected amounts are seeded from history and then frozen-with-a-nudge: never silent drift, only confirm-to-apply updates when actuals diverge.

The liquid **Projection** adds a **Typical spend** baseline so "what's left in checking" is realistic rather than systematically too high: per liquid Account, the trailing-three-month median of *non-recurring* `Net movement` (excluding template-matched occurrences and ad-hoc self-transfers), rendered as a *band*, not a point, so the curve advertises its own uncertainty. **Occurrence matching** — recognizing that a posted `JournalEntry` realizes a template, so the projected period switches from expected to actual — happens *at query time only*: there is **no FK from `JournalEntry` / `JournalLine` to a template**, so the ledger gains zero columns and templates can be added or deleted without touching a single posting. What-if **Scenario**s overlay ephemeral template overrides (toggle off / add / adjust amount) on the same computation and persist nothing.

Rejected: **scheduled future-dated postings** (recurring items that auto-post real `JournalEntries` ahead of time, reconciled on import) — the Firefly/GnuCash/YNAB norm, but it pollutes the ledger with speculation, needs amend/mismatch machinery when the bank's actual charge differs, and stretches the **Attach predicate** of ADR-0012 beyond self-transfers; this is the same approach ADR-0025 rejected for loans. **An explicit occurrence-link FK on the ledger** (tagging each realizing `JournalEntry` with its template at categorization time) — reliable, but a new ledger column and a new write step for a relationship that, unlike a loan payment's composite per-part attribution, is cheaply *re-derivable* from the mandate-backed matching key; kept as a deferred additive escape hatch if query-time ambiguity ever bites. **Detection-only templates** (no user confirmation) — the projection must never silently depend on a guessed pattern, and you cannot detect a subscription you only just signed up for. **A recurring-only projection** (no `Typical spend`) — makes "what's left in checking" read systematically high by pretending no money moves except tracked subscriptions. **A materialized projection or schedule table** — a cache with regeneration duties on every posting, edit, template change, and rate change, bought for queries that a pure function already answers.
