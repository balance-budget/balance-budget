# Frontend-to-API data layer: TanStack Query + OpenAPI codegen + branded IDs

The SPA at `src/Balance.Web.Client` talks to `Balance.Web` through a single conventional stack: **TanStack Query** (`@tanstack/react-query`) for all reads and mutations (caching, dedup, invalidation, optimistic updates), **wire types generated from OpenAPI at .NET build time** via `Microsoft.Extensions.ApiDescription.Server` + `openapi-typescript` (the document emits as a build artifact under `artifacts/` — no need to run the app — and `npm run build` runs codegen before `tsc -b`), **brand-at-the-boundary IDs** (generated types stay `string`, a thin `src/api/*.ts` layer wraps each fetch and re-brands using the existing `Brand<>` pattern), **`fast-json-patch`** to produce RFC 6902 patches for `PATCH` endpoints (compute `compare(original, edited)` against an editable copy so the dictionary-keyed lines from ADR-0005 work natively), and **per-component `isPending` / `isError` guards** with a shared `<Skeleton/>` + inline `<ErrorState/>` (no Suspense, no route-level error boundaries beyond what TanStack Router provides). The existing `src/lib/domain.ts` stops being a wire-shape mirror and becomes a pure view-model layer (e.g. `AccountSummary` with deterministic client-side `accentColor` / `iconName`); `src/api/*.ts` owns the wire boundary.

We rejected (a) TanStack Router loaders only because mutations (create/edit/delete JournalEntry) have no story other than imperative `fetch` plus manual `router.invalidate()`, and the dashboard's mix of overlapping queries (accounts, balances, register rows, KPI aggregates) wants real caching and dedup; (b) plain `fetch` + `useEffect` hooks because we'd own the loading/cache/error discipline ourselves and Strict-Mode double-fetch and stale-data bugs are exactly what TanStack Query already solves; (c) hand-written TypeScript wire types because that is the setup that already drifted in this codebase (`AccountSummary` carrying `balanceMinor` / `bankAccountNumber` / `accentColor` fields that the real `AccountOutput` never had); (d) generated types without brand-laundering because ADR-0004's parameter-swap safety stops at the C# boundary — `StronglyTypedId` flattens to `string` in the OpenAPI schema (via `TypedIdSchemaTransformer`) and an unbranded TS type would let an `AccountId` be passed where a `CounterpartyId` is expected; (e) customising OpenAPI emission to produce per-typed-id nominal schemas because the upstream cost (intercepting the schema transformer, keeping the names in sync) exceeds the cost of a one-line cast in the `api/*.ts` mapper; (f) a PUT-with-server-diff sibling for each PATCH endpoint because it would duplicate the entire update surface and risk the two diverging on the very immutability invariants ADR-0005 exists to enforce; (g) hand-written JSON Patch arrays at call sites because the per-line dictionary patches (`/lines/{lineId}/description`) are fiddly to type by hand and a `compare` over a snapshot is robust by construction; (h) `useSuspenseQuery` + route-level Suspense / error boundaries because route-level fallbacks are too coarse for the dashboard's per-widget streaming and the per-component-guard pattern stays grep-able. Auth and antiforgery are deliberately out of scope here — no endpoint currently calls `RequireAuthorization`, and `UseAntiforgery` does not fire on JSON minimal-API calls, so this layer assumes anonymous JSON access; both will be revisited when login lands.
