# `BankAccount` discriminates by `BankAccountType`; extractor dispatch is per-`BankAccount` `ImporterKey` via interface metadata

A second statement format (ING credit card) lands alongside the existing ING current-account format and forces the dispatcher work ADR 0010(j) explicitly deferred. We evolve `BankAccount` and the import surface together. `BankAccount` gains three columns: `Type` (`BankAccountType` enum: `Current`, `Savings`, `Card`; `NOT NULL`; default `Current`); `CardIdentifier` (nullable text, the masked card number normalised — uppercase, spaces stripped — e.g. `"1234************5678"`); and `ImporterKey` (nullable text, naming the extractor to use for future statement imports onto this BankAccount; same key namespace as the existing `BankTransaction.ImporterKey`). The two existing CHECK constraints are replaced by a per-`Type` identifier rule (`Current` → `Iban IS NOT NULL`; `Savings` → `Iban IS NOT NULL OR AccountNumber IS NOT NULL`; `Card` → `CardIdentifier IS NOT NULL`), the existing conditional-currency CHECK is retained, and a new CHECK enforces `Type='Card' ⇒ AccountId IS NOT NULL` so that counterparty BankAccounts can never be `Card`. `IBankTransactionExtractor` gains two instance members — `Key` (the same string namespace, e.g. `"Ing.CurrentAccount.V1"`, `"Ing.CreditCard.V1"`) and `SupportedType` (the `BankAccountType` the extractor knows how to ingest) — and `Balance.Integration.Ing/ServiceCollectionExtensions` registers each extractor against `IBankTransactionExtractor` as an enumerable. `BankTransactionImportService` is rewritten to take `IEnumerable<IBankTransactionExtractor>` and resolve the right one by `e.Key == bankAccount.ImporterKey`, returning `ErrorCodes.ImportBankAccountNotImportable` when `ImporterKey` is null and `ErrorCodes.ImportBankAccountWrongImporter` when the resolved extractor's `SupportedType` disagrees with the BankAccount's `Type`. The same `SupportedType` invariant is enforced at BankAccount write-time (creation and update reject incoherent `ImporterKey`/`Type` pairs), so the database never holds a row the dispatcher would have to reject. `IngCreditCardTransactionExtractor` is enhanced to populate `BankTransaction.CounterpartyAccountNumber = statement.Counterparty` for `CreditCardTransactionType.DirectDebit` rows only; merchant rows (`Payment`, `Receipt`, `CashWithdrawal`, `Fees`, `Correction`) continue to leave it null. With that single field populated, ADR 0013's Attach predicate clause (3) fires unmodified on the card-side pay-down row against the current-account self-transfer JE that the funding-account import created — completing the self-transfer affordance `CONTEXT.md` already promised for Current → Card pay-downs. The endpoint surface is unchanged: `POST /api/bank-accounts/{id}/imports` remains the single import route, with all dispatch consequences now folded into the service layer. A small registry endpoint (or DTO field on the BankAccount creation form) exposes the available `(Key, SupportedType)` pairs to the SPA so that the BankAccount creation UI can render the `ImporterKey` dropdown filtered by the chosen `Type` — derived directly from the same `IEnumerable<IBankTransactionExtractor>` the dispatcher consults, no parallel registry to maintain.

We rejected (a) reusing `AccountType` as the BankAccount discriminator because it collides head-on with the load-bearing ledger-side **AccountType** glossary entry (Asset / Liability / Equity / Income / Expense), and the two are conceptually orthogonal — a Card **BankAccount** belongs to a Liability **Account**, a Savings **BankAccount** to an Asset **Account** — so the overlap would generate steady confusion in both code and conversation. (b) Modelling cards as a sibling entity (`Card` next to `BankAccount`) because it doubles the import plumbing surface — separate extractor signature, separate endpoint, separate DI registration — for the same bookkeeping role, and the categorisation / attach machinery would have to be re-pointed at both entity types or hide a union behind it. (c) The status-quo "PAN goes in `AccountNumber`" coupling, because it lets the existing CHECK constraint pass for the wrong reason and silently bifurcates the meaning of one column across two bank-product shapes — a debt the credit-card extractor was already paying by abusing `bankAccount.AccountNumber` for card matching. (d) Storing only last-4 in `CardIdentifier` because two cards on the same owning **Account** ending in the same four digits would collide on import-time matching, and we lose the round-trip equality against the statement-header string the parser already emits. (e) A structured BIN + last-4 representation (separate columns) because it buys nothing beyond display until a second bank with a different masking style lands; the masked-as-given normalised string round-trips today and `CardIdentifier` can be split later without losing data. (f) Allowing `Type=Card` on counterparty BankAccounts because a counterparty's card details never appear on your statements — counterparties show up via merchant names or by IBAN, never by card number — and leaving the door open would invite unanswerable rows (a `Counterparty` BankAccount with a `CardIdentifier` we never observed). (g) Keeping the CHECK constraint loose ("at least one identifier set") and enforcing per-`Type` identifier rules in the service layer only, because it lets the database hold incoherent rows (a `Card` BankAccount with no `CardIdentifier`) and the schema and service-layer rules drift; the strict per-`Type` CHECK is the same posture ADR 0011 took for the conditional currency rule. (h) Deriving the extractor from `(BankName, BankAccountType)` because `BankName` is free-text and would need canonicalising before equality matching is safe, and format-version evolution (`Ing.CurrentAccount.V1` → `V2` during a CSV-format migration) has nowhere natural to live in such a derivation. (i) Caller-picks-at-upload (`StatementFormat` enum on the POST) because it re-asks the user something the system already knows after the first import, offers no mechanism to *prevent* picking the wrong importer, and forces every UI client to maintain its own importer catalogue. (j) Requiring `ImporterKey` on every owned BankAccount because creating a BankAccount you have not yet decided to import statements into should remain valid — the BankAccount is the bookkeeping anchor, the importer is a future-facing convenience — and the null-ImporterKey case is a meaningful "manual entry only" state. (k) Soft coupling — extractor self-validates only at import time — because an incoherent `ImporterKey`/`Type` pair would sit on rows until the first import surfaced it, and the UI could not filter the `ImporterKey` dropdown by the chosen `Type` without duplicating the mapping outside the extractor; declaring `SupportedType` on the contract is the smallest interface change that closes both gaps. (l) Keyed DI (`AddKeyedScoped<IBankTransactionExtractor>("Ing.CurrentAccount.V1")`) for the dispatch because the extractor's metadata (`Key`, `SupportedType`) then lives in DI registration strings separate from the type itself, and anywhere that needs the catalogue — the BankAccount creation UI being the primary case — would need a parallel registry mapping keys to types; `IEnumerable` + interface metadata puts both pieces on the type, one source of truth. (m) Populating `CounterpartyAccountNumber` on every credit-card row from `statement.Counterparty` because the funding-account IBAN is genuinely the counterparty only for `DirectDebit` (pay-down) rows — for merchant rows the counterparty is the merchant, not your own current account — and the lie would propagate into the Attach predicate, falsely firing a self-transfer hint for every routine purchase. (n) Promoting the card-to-funding-account link to a first-class `FundingBankAccountId` column on BankAccount because the link is already fully reconstructible from the `CounterpartyAccountNumber` on `DirectDebit` rows once that field is populated, and amending ADR 0013's Attach predicate to traverse a `FundingBankAccountId` would add machinery for a problem that the per-row counterparty number solves directly; if a future feature genuinely needs the link without consulting the BTs (e.g. displaying "this card is paid from {funding account}" before any pay-down has imported), the column can be added then without losing data. (o) Amending ADR 0013's Attach predicate clause (3) to also match by `BankAccountType=Card` against the owning **Account**, because the predicate as written already fires correctly on the card-side `DirectDebit` row once it carries the funding IBAN as `CounterpartyAccountNumber`; loosening clause (3) would weaken its precision (false-positive Attach hints are the worst categorisation-flow failure mode, called out by ADR 0013(e)) for benefit already achievable through extractor enhancement.
