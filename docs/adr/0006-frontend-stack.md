# Frontend stack: Vite + Tailwind v4 + TanStack Router

The SPA at `src/Balance.Web.Client` is built on Vite + React 19 + TypeScript with **Tailwind v4** (CSS-first `@theme` configuration via `@tailwindcss/vite`), **TanStack Router** for client-side routing (file-based routes under `src/routes/` via `@tanstack/router-plugin/vite`), **`lucide-react`** for icons, and **`@fontsource/poppins` + `@fontsource/jetbrains-mono`** for self-hosted webfonts. Design tokens (brand colors, surface alpha tints, foreground layers, semantic + category palettes, type scale, spacing, radii, blur, motion easings) live in a single `@theme` block in `src/index.css`, derived from the Balance Design System bundle. Component styles use Tailwind utilities directly; no CSS Modules and no separate `tailwind.config.js`. UI vocabulary in components matches `CONTEXT.md`: variables, props, and component names use `JournalEntry` / `Counterparty` / `BankTransaction` rather than the everyday-speech equivalents from the design prototype.

We rejected (a) plain global CSS with hand-written flat classes (the design system's native form) because it would have meant maintaining a parallel naming convention for every screen and forfeiting Tailwind's discoverability for the multi-screen surface we expect to grow; (b) Tailwind v3 with a `tailwind.config.js` because the design's tokens are already shaped as CSS custom properties and v4 lets them serve as both the design source-of-truth and the Tailwind theme without a JS mirror file; (c) CSS Modules per component because the styling burden of an admin-style dashboard does not justify the per-file class-name plumbing, and the components we want to ship lift cleanly from the design system's flat-class structure; (d) React Router over TanStack Router because TanStack's compile-time type-safe routes and natural pairing with TanStack Query (the likely future data-fetching layer once feature endpoints land) outweigh the larger ecosystem of React Router; (e) Lucide via CDN script tag (the design's substitution) because `lucide-react` ships tree-shakeable typed components that fit a Vite + TS build idiomatically; (f) Google Fonts `@import` because `@fontsource/*` self-hosts via npm with no external runtime request, no privacy surface, and Vite-managed fingerprinting at the cost of a few hundred KB checked into `node_modules`.

## Amendment (2026-05-20): Recharts for client-side charts

**Recharts** joins the stack as the charting library. Charts compose as React JSX (`<LineChart>`, `<Line>`, `<XAxis>`, …), which fits the rest of the codebase, and the library covers the chart types we expect in v1 (line / area / bar / pie). Stroke colours accept CSS custom-property strings verbatim, so the existing `visualHintFor` palette works without pre-resolution; custom tooltip and legend content are plain JSX, so Balance Design System integration stays local to the component.

We rejected (a) **Visx** because, while tree-shakeable and very flexible, composing scales / axes / tooltips by hand pays off only once a project has many bespoke charts — not for a single trend chart today; (b) **Nivo** because per-chart bundles run larger than Recharts and the project's maintenance pace has slowed; (c) **Tremor** because it ships a competing dashboard design vocabulary on top of the existing Balance Design System (it also wraps Recharts under the hood, so adopting it means doubling the abstraction layer for no extra capability we'd use); (d) **Apache ECharts** because at ~330kb gzipped and an imperative-options API it is overkill for a personal-finance dashboard; (e) **continuing with the hand-rolled SVG** because the chart is missing axes, labels, tooltips, and responsive sizing — adding all four ourselves duplicates what Recharts already provides.
