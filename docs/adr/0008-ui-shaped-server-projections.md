# Account list and Register are UI-shaped server projections

The `GET /api/accounts` list endpoint returns each `AccountOutput` enriched with the account's current `Balance` (a `Money` summed over its `JournalLines`) and a nullable `BankAccount` summary (`Iban?`, `AccountNumber?`, `Bic?`, `BankName?`) populated by joining the optionally-linked **BankAccount**; the single-account `GET /api/accounts/{id}/balance` endpoint stays for the cases that want only the balance. A new `GET /api/accounts/{id}/register?skip&take` endpoint projects the focal account's **JournalLines** into **RegisterRow**s server-side: each row carries the focal-account-signed `Money` amount (positive = money in, negative = out — *not* the raw debit/credit sign), the **JournalEntry** header (`Date`, `Description`, `CounterpartyId`/`CounterpartyName`), the focal line's `ReconciliationStatus` and `Description`, and the offsetting side as a list of `(AccountId, AccountName, Amount)` — one element for a simple two-leg entry, multiple for a split (e.g. one €100 purchase divided across `Groceries +60` and `Household +40`). The list is ordered `Date DESC, JournalEntryId DESC` and pages with `skip` / `take` like `ListJournalEntries`. The frontend `api/*.ts` mappers brand IDs and produce view-models directly from these shapes — no client-side joining, no client-side sign-flipping, no client-side split-handling.

We rejected (a) keeping `AccountOutput` bare and fetching balances per account because the dashboard sidebar renders the balance for every account and N+1 round-trips on first load is exactly the kind of cost a list endpoint exists to amortise; (b) a separate `GET /api/accounts/balances` aggregate alongside a bare list because it doubles the request count, forces a client-side join, and shows partially-balanced sidebars during streaming load with no offsetting benefit (no caller wants accounts without balances on any current screen); (c) fetching `/api/bank-accounts` separately and joining `AccountId → BankAccount` in TanStack Query because the relationship is at-most-one and intrinsic to the **Account**'s identity-on-the-dashboard, so the join belongs server-side rather than as `useBankAccounts()` plumbing repeated at every consumer; (d) deriving the register view from a generic `GET /api/journal-entries?accountId=` plus client-side collapsing because three concerns live naturally in the projection — the focal-account sign flip (debit-normal vs credit-normal **Accounts** present the same line with opposite UI signs), the split-handling shape (one row per focal line, offsetting side as a list), and account-name enrichment for the offsetting accounts — and pushing any of them into TypeScript means every consumer of the register has to repeat the same logic; (e) exploding splits into multiple rows on the focal account because the focal-amount column would then disagree with what a bank statement shows for the same entry, which is exactly what reconciliation against a **BankTransaction** needs to compare. Adding `Balance` to the list endpoint and projecting a `RegisterRow` shape are deliberate trade-offs: the **AccountOutput** is no longer a thin mirror of the **Account** entity (it carries a derived `Balance` and a joined `BankAccount` summary) and **RegisterRow** is a view-only concept that never round-trips back to the server. We accept that coupling because the alternative — preserving "endpoint = entity" purity and pushing the per-account view together in TS — has been tried implicitly by the current `lib/domain.ts` and produced exactly the drift this ADR exists to undo.
