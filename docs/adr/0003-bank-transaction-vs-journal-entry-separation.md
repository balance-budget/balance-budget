# `BankTransaction` (immutable import fact) is separate from `JournalEntry` (revisable interpretation)

Imported bank-statement rows are stored in a `BankTransaction` table — immutable, deduplicated by hash, carrying the raw imported amount, dates, description, and `FromBankAccountId` / `ToBankAccountId` (resolved against the unified `BankAccount` registry). The ledger interpretation lives in `JournalEntry` (with one or more `JournalLines`), which may reference a `BankTransaction` via `JournalEntry.BankTransactionId?` when derived from an import, or leave it null for cash entries.

We chose this split — uncommon in personal-finance software but standard in ERP / professional bookkeeping (SAP "Document" → "Journal Entry") — because it (a) lets re-imports be idempotent via the hash, (b) lets users revise categorisation, split, or merge entries without losing the original bank record, (c) keeps audit trail intact, and (d) cleanly separates "what the bank reported" from "what we did about it". The name `BankTransaction` (matching ISO 20022 vocabulary) was preferred over `Transaction` to avoid collision with database transactions and the everyday meaning of the word.
