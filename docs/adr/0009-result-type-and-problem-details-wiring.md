# Service errors are returned as `Result<T>`, not thrown as `DomainException`

Service methods return `Result<T>` (or non-generic `Result` for side-effect-only operations like `DeleteAsync`). `ServiceError` is a sealed hierarchy living in `Balance.Services.Contracts` — `NotFoundError(Entity, Id)`, `ConflictError(Code, Message)`, `InvariantError(Code, Message)`, `ValidationError(IReadOnlyDictionary<string, string[]>)` — each carrying a stable `Code` string that becomes the `code` extension on the wire `ProblemDetails`. The FluentValidation endpoint filter is unchanged and continues to short-circuit with `Results.ValidationProblem` before the handler runs; `JsonPatchExtensions.ApplyAndValidateAsync` is changed to return `Result<T>` so PATCH endpoints chain patch-apply → service in one Result pipeline. Endpoints map results via typed extensions in `Balance.Web/Mappers/ResultExtensions.cs` (`.ToOk<T>()`, `.ToCreatedAt<T>(...)`, `.ToNoContent()`, …) — each returning a typed `Results<TSuccess, NotFound<ProblemDetails>, Conflict<ProblemDetails>, UnprocessableEntity<ProblemDetails>, ValidationProblem>` union so the minimal API source generator populates OpenAPI response shapes automatically. Save-time race backstops live in one place: a `DbContextExtensions.SaveAsync` helper catches `UniqueConstraintException` (→ `ConflictError("uniqueness_conflict", …)`) and `ReferenceConstraintException` (→ `ConflictError("referenced", …)`) via the `EntityFrameworkCore.Exceptions` package; pre-flight `AnyAsync` uniqueness checks stay in each service as the user-facing-message path and remain the common case. Every other `DbUpdateException` keeps propagating to a 500 — that is a bug, not a domain error. `DomainException`, `DomainExceptionKind`, and `DomainExceptionHandler` are deleted; the only remaining `IExceptionHandler` need is the default ProblemDetails middleware's 500-for-unhandled-exceptions behaviour, which `AddProblemDetails()` already provides.

We rejected (a) keeping `DomainException` + a cross-layer `IExceptionHandler` because 404/409/422 are normal outcomes of validating user input, not exceptional control flow — and because the `DomainExceptionKind` enum living in `Balance.Data` and being switched on in `Balance.Web` was a layering smell that needed a third file (the handler) to bridge, while FluentValidation already short-circuited cleanly without a throw, so two structurally identical 4xx response shapes were produced by two different mechanisms; (b) a flat `Result<T, Error>` where `Error` is a single struct carrying `Kind` + `Message` + optional `Errors` dict — i.e. `DomainException` minus the throw — because it leaves "currency not defined" and "account not found" distinguished only by a message string, which is exactly the information loss at the API boundary that motivated the move and which a `code` extension on `ProblemDetails` solves directly; (c) per-operation discriminated unions like `CreateAccountResult = OneOf<AccountOutput, NameTaken, CurrencyNotFound>` because the type surface explodes — every service method needs its own union and every endpoint needs its own exhaustive switch, and adding a new failure mode ripples through both layers, when a small closed hierarchy with `Code` strings registered in a central constants file gives 90% of the discrimination at a fraction of the cost; (d) a third-party library like `ErrorOr`, `OneOf`, `FluentResults`, or `LanguageExt` because the public service contract would be permanently tied to a transitive type from outside `Directory.Packages.props`'s deliberately minimal dependency surface, and the in-house Result is ~80 lines of `readonly record struct` + sealed records that needed no extension after the initial design; (e) dropping the pre-flight `AnyAsync` uniqueness checks and relying solely on catching `UniqueConstraintException` from `SaveChangesAsync` because the user-facing path then surfaces "a conflicting record already exists" instead of the specific "An account named 'Groceries' already exists", and a constraint-name switch in the catch (`IX_Accounts_Name → "account.name_taken"`) recreates exactly the "branch on index name" coupling the previous decision moved away from — just at a typed-exception layer instead of regex on `InnerException.Message`; (f) accumulating errors across all checks within one service call so the client sees every problem at once, because most service-level checks are deeply sequential (you cannot validate line amounts against an account whose existence has not been confirmed, and "lines[2].amount is zero on a non-existent account" is a misleading error to surface), while the within-DTO shape errors that *do* benefit from accumulation are already aggregated by the FluentValidation filter at a different layer; (g) returning bare `TypedResults.NotFound()` / `Conflict()` / `UnprocessableEntity()` and relying on `AddProblemDetails()` + `UseStatusCodePages()` to fill the body at runtime because the OpenAPI spec then says "404 with no body" while the wire response carries a `ProblemDetails`, dropping both the `code` extension that clients branch on and the entity/id detail fields that fix complaint (b). The trade-off accepted is that the typed-union return signatures are slightly over-broad — a `GetById`-style read that only ever produces success-or-not-found still has `Conflict`/`UnprocessableEntity`/`ValidationProblem` slots in its union, unless the endpoint reaches for `.ToOkOrNotFound<T>()` specifically — and adding a new `ServiceError` variant requires touching the central `Map` switch in `ResultExtensions`. Both costs are bounded to single files; the alternative was distributing the same coupling across every service and every endpoint.
