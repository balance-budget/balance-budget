# Currency code and minor-unit scale are immutable after creation

A **Currency**'s `Code` and `MinorUnitScale` are fixed once the row exists; only `Name` and `Symbol` can be edited (the update path on `ICurrencyService`/`CurrencyEndpoints` carries only those two). To correct a mistyped code or scale you delete the currency and recreate it, which is safe because delete is only permitted while the currency is unused. The currency management screen (`/settings/currencies`) presents `Code` and `MinorUnitScale` as read-only in the edit form and hints on create that scale cannot be changed later.

The scale is the interpreter for every stored amount, and amounts in a currency live in more places than its foreign keys. `Account.CurrencyCode` and `BankAccount.CurrencyCode` are real FKs (so the delete guard counts them and stays in lockstep with the DB `RESTRICT`), but `BankTransaction.ForeignAmount`/`ForeignCurrencyCode` store amounts in a currency with **no FK and possibly no account at all** — an FX leg can hold USD amounts while zero USD accounts exist. Allowing a scale edit would therefore need a *stricter* "no amounts anywhere" scan than the delete guard, and silently reinterpret immutable audit data if it ever missed a row (a stored `1050` flipping from $10.50 to $1050 when scale 2 → 0).

We rejected a mutable scale gated on a broader usage scan: the only legitimate need is fixing a typo on a freshly-created currency, which delete-and-recreate already serves, and a second guard that differs from the delete guard is machinery and cognitive load for a near-nonexistent case. New currencies (incl. crypto) are still added freely by inserting a row; this ADR only constrains *editing* an existing one.
