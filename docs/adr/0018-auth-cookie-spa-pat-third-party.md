# Authentication: cookie for SPA, opaque per-user PATs for third parties

ASP.NET Core Identity is the user store and password authority; the SPA authenticates with a same-origin cookie, and third-party callers (CLI scripts, importers, cron jobs) authenticate with opaque per-user **Personal Access Tokens** (PATs) of the shape `bal_pat_<32-byte-base64url>`. The two schemes coexist via a default **policy scheme** that sniffs the `Authorization` header: if it starts with `Bearer bal_pat_` the request is forwarded to the API-token handler, otherwise to the cookie handler. A single `RequireAuthorization()` policy on the `/api` group covers every endpoint with carve-outs (`AllowAnonymous`) for the setup wizard, the login endpoint, healthz, OpenAPI, and Scalar docs. CSRF protection uses the **canonical ASP.NET antiforgery pattern**: `IAntiforgery.GetAndStoreTokens` writes a HttpOnly *cookie token* as a side effect and returns the matching *request token* in the body of `GET /api/antiforgery/token`; the SPA caches that request token in JS memory (not localStorage — XSS-exfiltratable) and echoes it as the `X-XSRF-TOKEN` header on every mutation. The browser sends the cookie token automatically; the server validates that the cookie token and header token form a matched pair. The antiforgery filter skips when the authenticated scheme is the API-token scheme (CSRF is a browser-only concern; a request that authenticates via `Authorization: Bearer …` has no ambient credential to forge).

Identity is wired as `AddIdentityCore<TUser>().AddSignInManager().AddEntityFrameworkStores<BalanceDbContext>()` — the *core* flavour, not the full `AddIdentity` or the UI-bundled `AddDefaultIdentity`. The Identity tables live in `BalanceDbContext` (already implementing `IDataProtectionKeyContext`, so this context already hosts non-domain infrastructure tables); migrations ship through the existing `Balance.Data.Sqlite` / `Balance.Data.PostgreSql` assemblies. The `TUser` is `BalanceUser : IdentityUser<UserId>` where `UserId` is a `record struct UserId(Guid Value)` generated by the same source generator as every other typed ID in the codebase (ADR 0004), and the Identity join tables (`IdentityUserClaim<UserId>`, `IdentityUserLogin<UserId>`, `IdentityUserToken<UserId>`) carry the typed key through with value-converter wiring on `OnModelCreating`. `RoleManager` is not registered — there are no roles (ADR 0017). The cookie is configured `HttpOnly`, `Secure` in production, `SameSite=Strict`, sliding 30-day lifetime; `SecurityStampValidationInterval` is reduced from the 30-minute default to ~5 minutes so that disabling a user (via `LockoutEnd`) takes effect on existing SPA sessions within minutes rather than half an hour. The password policy is length-first (minimum 12, no composition rules) per current NIST SP 800-63B guidance. The API-token handler resolves the bearer string by SHA-256-hashing the random portion and looking it up against `ApiToken.TokenHash`; the row carries a displayable `Prefix` and `Last4` for the management UI but no recoverable copy of the secret, and the handler additionally rejects tokens whose owner's `LockoutEnd` is in the future (so user disable is effective for tokens immediately, not on the next stamp cycle).

We rejected, for the setup wizard specifically, (i) the empty-table check alone without a setup token, because the bootstrap window is real ("instance is up, first user not yet created") and on a publicly-reachable deployment that window is the moment of maximum exposure — a single env var closes it without coupling the app to a specific hosting topology; (ii) a localhost-only setup wizard refusing non-loopback requests, because that encodes a "you must SSH-tunnel to bootstrap" assumption that doesn't belong in the application layer (the reverse proxy is the right place for that opinion, if anyone wants it) and makes the first-deploy experience meaningfully worse; (iii) an IP allow-list, for the same reason. For the broader auth surface we rejected (a) `MapIdentityApi<TUser>()`, the .NET 8 scaffolded auth endpoints, because they bundle a public `/register` route (we have no public signup — ADR 0017's provisioning is setup-wizard-then-admin-UI) and their bearer-token mode uses opaque short-lived tokens stored in `AspNetUserTokens` with refresh-token rotation — that shape is *session-via-bearer* for SPAs, not *named long-lived API key for a cron job*, and using it for the third-party use case would leave us with two parallel token implementations anyway; (b) `AddDefaultIdentity` and its Razor UI, because the SPA owns every non-`/api` URL and shipping Razor pages we never render is dead weight; (c) JSON Web Tokens (signed JWTs) for the PAT format, because long-lived API keys want instant revocation and JWTs only revoke by either waiting for expiry or maintaining a deny-list — the deny-list collapses the no-DB-hit benefit that's JWT's only real advantage, and there is no natural surface to *name* a JWT the way users actually want to name their tokens ("ING importer cron"); (d) Argon2id / PBKDF2 for `ApiToken.TokenHash`, because password KDFs exist to slow down offline brute-forcing of low-entropy human-chosen secrets — PATs are server-generated with 256 bits of entropy, so a fast cryptographic hash (SHA-256) gives the same security at a tiny fraction of the per-request CPU cost, matching how GitHub stores PATs; (e) per-endpoint multi-scheme attributes (`AuthenticationSchemes = "Cookie,ApiToken"`) instead of a policy scheme, because that pattern duplicates the scheme list on every endpoint, makes the default policy useless, and obstructs adding a third scheme later (e.g. OIDC for a hypothetical mobile client); (f) separate URL prefixes for cookie vs token endpoints (`/api/...` cookie-only, `/api/external/...` token-only), because the vast majority of endpoints are reachable from both an SPA session and a CLI/cron script, and doubling the endpoint surface for a notional separation that the policy scheme handles for free is bad value; (g) `SameSite=Lax` instead of `Strict`, because the canonical "Strict breaks links from email" argument does not apply here — this is a self-hosted personal-finance app with no marketing email deep-links, the only entry points are bookmarks and typed URLs (which are top-level user-initiated navigations and *do* receive Strict cookies), and the SPA owns routing once you're in; the marginal usability cost of an Gmail-link click landing on the login screen is acceptable in exchange for closing virtually all browser-mediated CSRF vectors at the cookie layer (antiforgery stays as defense-in-depth); (h) requiring antiforgery on the login endpoint itself ("login CSRF"), because the threat model — attacker tricks victim into authenticating as the attacker's account so subsequent victim activity is logged against the attacker — is irrelevant when registration is closed (ADR 0017) and there is no scenario where a victim would unwittingly want to "be" a different already-existing user.

The provisioning flow is two-stage and intentionally narrow: a one-shot `POST /api/auth/setup` (anonymous) that is gated on **both** `AspNetUsers` being empty **and** the request presenting a deploy-time setup token matching `Auth:SetupToken` (read from configuration / env var), and that self-disables on the first successful call by virtue of the empty-table check. After the first user exists the endpoint returns `404` — not `409`, not "already set up" — so probing the URL does not reveal that the setup machinery ever existed. Subsequent users are added through the authenticated `/api/admin/users/*` surface in the SPA. There is no `/api/auth/register` endpoint and no CLI command for user creation — the setup wizard removes the cold-start problem and the in-app UI handles the warm path, so neither plaintext passwords in `appsettings` nor an SSH-and-run-CLI ritual is part of the surface. The setup token exists specifically to close the bootstrap-window race ("someone else hits `/api/auth/setup` before I do, creates user #1, and now owns the ledger"), and is layered on top of — not in place of — the empty-table check so a token leak after deploy still cannot create a user once one exists. The PAT minting endpoint (`POST /api/admin/tokens`) returns the plaintext token *once* in the response body; the value is never recoverable afterwards. Token rows carry an optional `ExpiresAt` (default null = no expiry, matching GitHub PAT semantics) and a `RevokedAt` for explicit revoke without delete (so listing past tokens for audit remains possible — "the cron token I revoked last month was last used three days before I revoked it").

The accepted costs of this design are: (i) `BalanceDbContext` grows a dependency on `Microsoft.AspNetCore.Identity.EntityFrameworkCore` (small architectural smell, but the precedent of `IDataProtectionKeyContext` already exists); (ii) the typed `UserId` wiring requires a one-time `OnModelCreating` block for the Identity join tables (one-off cost, paid in exchange for ADR 0004 consistency); (iii) the policy scheme makes the "what scheme is this request" answer indirect — debugging an auth failure means checking the forward logic before the handler — which is the usual cost of any sniff-and-route pattern; (iv) the load-bearing-ness of the `bal_pat_` prefix means a user pasting a PAT without the prefix will be silently routed to the cookie scheme and rejected as "not logged in", which is a confusing failure mode that the API-token issuance UI must compensate for by always including the prefix in the displayed copy-paste value.
