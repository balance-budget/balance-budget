# `BankTransaction` promotes SEPA / FX fields to columns and carries the remainder in a typed key-value `BankTransactionMetadata` side-table



Bank statements carry more useful structured information than the universal `Description` / `CounterpartyName` / `CounterpartyAccountNumber` columns ADR 0010 promoted; ING CSV alone parses `MandateId`, `Reference`, `ValueDate`, an FX block, an internal transaction code, a SEPA creditor, a card sequence, and more. We widen `BankTransaction` along two axes. First, a fixed bank-agnostic SEPA / ISO-20022 column set is promoted onto the entity — `ValueDate?`, `Reference?`, `MandateId?`, `SepaCreditorId?`, `ForeignAmount?`, `ForeignCurrencyCode?`, `ExchangeRate?` — because these recur across European bank exports, are the fields most useful for everyday display and for upcoming categorisation-matching rules, and benefit from being first-class indexable columns on the row itself. Second, anything an extractor parses that is not promoted lives in `BankTransactionMetadata`: two normalised tables `BankTransactionMetadataKey` (`Id`, `Name UNIQUE`) and `BankTransactionMetadataValue` (`BankTransactionId`, `KeyId`, `StringValue text NULL`, `IntegerValue bigint NULL`, `CHECK ((StringValue IS NULL) <> (IntegerValue IS NULL))`, `PRIMARY KEY (BankTransactionId, KeyId)`). Keys are a single global namespace owned across all extractors — bank-agnostic names where the concept is shared (`SepaCreditorName`, `OtherParty`), bank-prefixed only for genuinely bank-specific extras (`IngTransactionCode`, `IngMutatiesoort`, `IngTag`). Nested values flatten with dotted keys (`ForeignMarkUp.Amount` + `ForeignMarkUp.CurrencyCode`, `CardSequence.Number` + `CardSequence.Total`). Currency amounts and counts use `IntegerValue` so minor-unit values stay queryable without per-key parse rules; free-text values use `StringValue`. `BankTransaction` also gains an `ImporterKey` column identifying the extractor that produced the row (e.g. `Ing.CurrentAccount.V1`; null for manually-created rows), so that a one-shot **idempotent backfill** can run from `RawSource` for historical rows: a startup task in `Balance.Services` invoked after `MigrateDatabaseAsync` dispatches each non-manual row to its named extractor, replaces its `BankTransactionMetadata` set, and fills the new promoted columns. The same code path is reused whenever an extractor's parsing improves — `BankTransactionMetadata` is the only post-import-mutable surface on a `BankTransaction` other than the existing `Dismissed*` user fields, and it is rebuilt wholesale rather than diff-merged. API surface: the detail endpoint exposes the promoted columns as ordinary fields plus a typed list `[{ key, stringValue?, integerValue? }, ...]`; the list endpoint stays unchanged (the promoted columns already give the Inbox the SEPA fields worth scanning, and loading the blob for every page is wasteful). Scope of this change is **plumbing only** — the data is captured, displayed in the detail view, and structurally ready for matching rules to consume, but no matching engine ships in this slice.

We rejected (a) a JSON `jsonb` / dict blob column on `BankTransaction` for the non-promoted fields because "list every distinct key ever seen across all rows" — the natural backbone of any future filter UI — is awkward on SQLite (and per-provider on PG), because flat indexes on a specific `(Key, Value)` lookup require expression indexes that are clean on PG but ugly on SQLite, and because the integer type for currency minor-unit values would collapse to strings at the OpenAPI boundary forcing per-key SPA-side reparse rules. (b) Per-bank metadata tables (`IngBankTransactionMetadata` joined 1:1) for the same reason ADR 0010(f) originally rejected them: schema sprawl per supported bank, joins on every read, no query benefit once a bank-agnostic shape exists. (c) A single `Value text NOT NULL` column on the value table with a `ValueType` enum discriminator, because two nullable columns with a single CHECK constraint is materially cleaner storage and preserves integer queryability natively — provider-uniform without enum-driven branching at read time. (d) Typed-per-bank JSON blobs (`IngImportMetadata` with named properties exposed verbatim in OpenAPI) because the wire shape would diverge per bank and the SPA would need discriminator-aware rendering for what is fundamentally a key-value bag. (e) Bank-namespacing every metadata key (e.g. `Ing.MandateId`) because SEPA / ISO-20022 concepts genuinely recur across European banks, and a flat namespace lets cross-bank queries (and the categorisation rules that will come later) match without first knowing which extractor produced the row; bank-prefixing is reserved for genuinely bank-internal extras. (f) A flat single-table KV without the `BankTransactionMetadataKey` lookup, because the keys form a small slowly-growing vocabulary reused across many rows; normalising saves space, removes the need for a `SELECT DISTINCT Key` aggregation when a future UI wants to enumerate the catalogue, and gives keys a stable surrogate ID that survives renames. (g) Leaving historical rows with empty `BankTransactionMetadata` (or only re-extracting lazily on detail-view read) because the primary use case for promoting these fields — categorisation matching — needs historical rows populated to be useful; the only reason `RawSource` was preserved by ADR 0010 was exactly this kind of re-extraction. (h) Wipe-and-reimport as the migration strategy, because the row-hash idempotency contract from ADR 0010 already supports running re-extraction repeatedly without data loss, and a re-runnable startup task doubles as the mechanism for shipping extractor improvements to existing rows. (i) Including `BankTransactionMetadata` in the list endpoint payload because the SEPA columns already cover Inbox scanning, the blob is by definition "everything not promoted", and an N+1 fetch (or a heavy JOIN-with-grouping) per Inbox page is wasted work for fields the user only consults on detail. (j) Shipping a matching / rules engine in the same slice as the plumbing, because matching has its own non-trivial design questions (precedence, conflict resolution, auto-apply vs suggest, scope-per-account-or-global) that deserve their own grilling pass and would balloon this change. (k) Promoting a wider column set covering FX markup / fee and SEPA creditor name + address, because the promoted set is sized to "fields a user would scan in the Inbox row or filter on directly"; markup / fee / creditor name live in the metadata blob where they remain retrievable and renderable on the detail view without inflating the row.
